## 3. automisc 失败根因（hex 实证）

### 3.1 LFH vs CDH flag_bits 对比

| entry | LFH flag_bits | CDH flag_bits | 矛盾? |
|---|---|---|---|
| `asd/` | `0x0800` (bit0=0) | `0x0800` (bit0=0) | OK |
| `asd/good-已合并.jpg` | `0x0800` (bit0=0) | `0x0801` (bit0=1) | 🔥 **不一致** |
| `asd/qwe.zip` | `0x0800` (bit0=0) | `0x0800` (bit0=0) | OK |

**矛盾点**：加密的 entry 在 LFH 中标志位是 0（明文），但 CDH 中标志位是 1（加密）—— **CDH 假加密**（形态 B）。

### 3.2 zipfile 的判定逻辑

`zipfile.ZipFile(...)` 读 `info.flag_bits` 走 **CDH**（central directory header，决定 entry 的"官方属性"）：

```python
>>> import zipfile
>>> with zipfile.ZipFile(zp) as zf:
...     for info in zf.infolist():
...         print(info.filename, hex(info.flag_bits))
asd/                                     0x0800
asd/good-已合并.jpg                      0x0801   ← zipfile 看到的"加密"
asd/qwe.zip                              0x0800
>>> zf.extractall('/tmp/test')           # → RuntimeError: ...is encrypted
```

**zipfile 看 CDH 判加密 → 报"is encrypted" → 失败**。

### 3.3 automisc `_is_pseudo_encrypted` 漏识别

源码 `src/automisc/core/actions/zip_chain.py:30-61`：

```python
def _is_pseudo_encrypted(zip_path: Path) -> bool:
    i = 0
    while i < len(data) - 4:
        if data[i:i+4] == b"PK\x03\x04":  # ← 只扫 LFH
            flag = struct.unpack("<H", data[i+6:i+8])[0]
            if not (flag & 0x1):
                return False  # ← 形态 B: LFH bit0=0 直接被杀
```

**漏点**：完全没看 CDH！zipfile 读 CDH 判定加密，automisc 却只读 LFH 判定 → **判定错位**。

### 3.4 链 DAG 流转

```
1) try_unzip fail (zipfile 看 CDH bit0=1)
   → _is_pseudo_encrypted 检查 → 看到 LFH bit0=0 → is_pseudo=False
   → 设 encrypted=True, stop_reason="真加密 zip, 需密码 (chain 立刻停)"
2) chain 立刻停（per v0.5-journal-highlight-keywords）
3) fix_pseudo_encryption 根本不跑
4) 解压失败 → Owner 报告 bug
```

## 4. AI 兜底（per §5.3 step 3）

### 4.1 writeup 提示

> 写 wp 的师傅说「直接将加密位改成 00 00」——没说改哪里。**实际他改的是 CDH 中的加密位**（因为 zipfile 读 CDH 判加密）。

### 4.2 手动修复（hexedit / 010editor）

```bash
# 找到 CDH 标志位 (offset 8, 2 bytes)
$ xxd -s 0x22 -l 50 00000038.zip     # 0x22 = LFH 偏移
00000022: 6c66 8b41 0008 0000 0000    # CDH 签名前

# CDH entry (PK\x01\x02) 在 zip 末尾 EOCD 之前
# 找到 `asd/good-已合并.jpg` 对应的 CDH, 把 offset 8 的 01 改成 00
$ xxd 00000038.zip | grep -A1 "good-已合并"
# → 在 PK\x01\x02 后第 8-9 字节, 01 00 → 00 00
```

修完后 `unzip` / `zipfile` 都直接解。

### 4.3 automisc 应该自动做这件事

**修复**：`FixPseudoEncryptionAction` 现有修复代码（line 152-167）**已经会同时修 LFH + CDH**——只要 `_is_pseudo_encrypted` 放它过就行。

**最小修复** = 升级 `_is_pseudo_encrypted` 同时扫 CDH 标志位。

## 5. 伪加密的 3 种形态（架构升级输入）

| 形态 | LFH bit0 | CDH bit0 | 旧代码 | 新代码 | 触发源 |
|---|---|---|---|---|---|
| **A** | 1 | 0 | ✅ | ✅ | 工具栏手动改 LFH |
| **B** | 0 | 1 | ❌ | ✅ | **owner 2026-06-20 命中**（最常见，foremost/工具雕的产物）|
| **C** | 1 | 1 | ✅ | ✅ | v0.5-train-002-175b 的现有 fixture |

**zip 标准规定**：LFH 和 CDH 都要有加密位（一致性校验）。**实际伪造**通常改一处即可——zipfile 读 CDH 判定，所以**形态 B**（只改 CDH）是最常见的"工具友好型"伪加密。

---

## 6. automisc 应补能力（架构升级输入）

| 能力 | 现状 | 应补 |
|---|---|---|
| **`_is_pseudo_encrypted` 扫 CDH** | 只看 LFH | **同时扫 CDH**，任一 bit0=1 且无真加密 header → 伪加密 |
| **`FixPseudoEncryptionAction` 修 CDH** | **已经修 LFH + CDH** | ✅ 不用改（修代码对，缺的是检测）|
| **fix_pseudo 跨形态覆盖率** | 单测只覆盖形态 C | 加形态 A / B 单测 fixture（3 形态都覆盖）|
| **journal 归集伪加密形态** | 只说"伪加密" | 加 "pseudo_form" 字段 (A/B/C) → 训练统计 |

## 7. 架构判定（per §5.2）

| 判定项 | 结论 |
|---|---|
| 同类样本 | **≥ 3 类**：形态 A / B / C 都是「伪加密」模式变体（**不是 3 道题**）|
