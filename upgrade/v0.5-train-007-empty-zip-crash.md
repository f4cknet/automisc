## 2. Bug 1 (高优先级) — 空 ZIP 崩溃

### 2.1 复现

```python
import zipfile, tempfile
from pathlib import Path
from automisc.tools.misc.archive.zip_classify import ZipClassifyAdapter

empty = Path(tempfile.mktemp(suffix='.zip'))
with zipfile.ZipFile(empty, 'w') as zf:
    pass  # 0 entry

a = ZipClassifyAdapter()
result = a.run(str(empty))
# UnboundLocalError: cannot access local variable 'verdict_summary'
#                     where it is not associated with a value
```

### 2.2 根因

`src/automisc/tools/misc/archive/zip_classify.py:109-127` 的 4 个 if/elif 分支：

```python
if n_pseudo > 0 and n_real > 0:    # 0 0 0 → False
    verdict_summary = ...; action = ...; severity = ...
elif n_pseudo > 0:                  # 0 → False
    verdict_summary = ...; action = ...; severity = ...
elif n_real > 0:                    # 0 → False
    verdict_summary = ...; action = ...; severity = ...
elif n_clear > 0:                   # 0 → False
    verdict_summary = ...; action = ...; severity = ...
# 全部跳过 → verdict_summary / action / severity 未绑定
```

后续 line 157 `f"{verdict_summary}\n\n..."` 引用 → **UnboundLocalError**。

### 2.3 修复

加 `else` 兜底分支：

```python
else:
    # 空 ZIP (合法 zip 但 0 entry) — 兜底防 UnboundLocalError (per v0.5-train-007)
    verdict_summary = "空 ZIP: 0 entry (合法 ZIP 但无内容)"
    action = "无需操作 (ZIP 不含任何 entry)"
    severity = 1
```

### 2.4 单测

`tests/unit/tools/misc/archive/test_zip_classify.py::test_empty_zip_no_unbound_local_error`:
- 构造 0 entry ZIP
- 跑 `ZipClassifyAdapter().run()`
- 断言: 不崩溃 + exit_code 0 + verdict SP severity 1 + "空 ZIP" 文本

**单测 14/14 通过** (新增 1 个)。

## 3. Bug 2 (中优先级) — algorithm 边界风险: data descriptor

### 3.1 问题描述 (per Owner 反馈)

`_classify_zip_entries` 强依赖 LFH 里的 `comp_size` 和 `crc` 取数据:

| 行号 | 用法 |
|---|---|
| `src/automisc/core/actions/zip_chain.py:75` | `comp_size = struct.unpack("<I", data[i + 18 : i + 22])[0]` — 从 LFH 取 comp_size |
| `src/automisc/core/actions/zip_chain.py:166` | `crc = struct.unpack("<I", data[lfh_offset + 14 : lfh_offset + 18])[0]` — 从 LFH 取 crc |

**如果 ZIP 使用 data descriptor** (LFH bit 3 = 1), writer 写 LFH 时还**不知道** comp_size / crc, 这两个字段被填 0, 真值在 LFH 之后 (data 区后) 的 PK\x07\x08 data descriptor 里。

**后果**: 我的代码会拿到 comp_size=0, crc=0 → `file_data = b""` → `_check_pseudo` / `_try_inflate_with_crc` 行为异常。

### 3.2 Owner 决策点

> "如果你要, 我下一步可以直接给出一个最小修复补丁:
> 1. 修 empty zip 崩溃 ✅ done
> 2. 补一条单测 ✅ done
> 3. 再评估要不要扩 data-descriptor 支持"

**我的判断**: **暂不扩** — data descriptor 在 CTF 中**不常见**, 没看到 owner 实战样本命中这个 case. 扩 data descriptor 需要:
- 检测 LFH bit 3
- 在 `data_start + comp_size_estimated` 之后找 PK\x07\x08
- 解析 data descriptor 拿真 comp_size / crc
- 处理 Zip64 变体 (data descriptor 可以是 24 字节含 8-byte size)

按 AGENTS.md §5.2 「架构判定标准」: 需 ≥ 3 同类样本才升架构. 当前只有 owner 指出风险, **没有** 实战样本命中. **defer**, 等实战命中再升架构.

**兜底**: 当前实现若遇到 data descriptor 仍可正常运行, 只是会保守归 real (per `_check_pseudo` 失败 + `_try_inflate_with_crc` 失败 → real 默认分支). 不会误导用户跑 ZipCenOp r, 只是会建议 bruteforce. **安全兜底**.

## 4. Bug 3 (低优先级) — algorithm 边界风险: 非 store/deflate method

### 4.1 问题描述

`_check_pseudo` 只处理 method 0 (stored) 和 method 8 (deflate):

```python
# src/automisc/core/actions/zip_chain.py:137-149
def _check_pseudo(file_data, expected_crc, method) -> bool:
    if method == 0:
        return (zlib.crc32(file_data) & 0xFFFFFFFF) == expected_crc
    if method == 8:
        return _try_inflate_with_crc(file_data, expected_crc)
    # 其他 method (bzip2, LZMA, zstd, xz, ...): 保守返回 False
    return False
```

**非 store/deflate** method (12=bzip2, 14=LZMA, 93=zstd, 95=xz) → `_check_pseudo` 返回 False → 走到默认 real 分支.

**后果**: 非 store/deflate method 的伪加密 zip 会被判 real (建议 bruteforce, 但实际是 flag 改了的明文).

### 4.2 Owner 决策点

按 AGENTS.md §5.2 同 §3, **暂不扩**. CTF misc 中非 store/deflate method **罕见**. 扩需要:
- method 12 → bzip2.decompress
- method 14 → lzma.decompress (header 解析)
- method 93 → zstandard (需第三方库)
- method 95 → xz (需 lzma)

引入第三方依赖 (zstandard) 不在 automisc 当前依赖列表, 不建议.

**兜底**: 同 §3, 当前保守归 real 不误导用户跑破坏性 ZipCenOp r.
