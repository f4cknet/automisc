# v0.5-binwalk-extract — binwalk adapter 补 PEM 关键字 + 自动 binwalk -e 提取

> **创建时间**：2026-06-15 23:14
> **状态**：🟡 design（实施中）
> **触发**：Owner 2026-06-15 实测 — `binwalk -e greatescape.pcap` 直接解出 2 个 PEM private key (offset 0x51D60 + 0x52A7D, 3273 字节 each)，但 automisc binwalk adapter 报"0 suspicious point(s)"
> **Owner 决策**：B 方案（关键字 + 一键提取 + 同目录输出）
> **关联迭代**：v0.5-pcap-protocol-router（greatescape 同一题暴露的姊妹 bug）
> **推翻旧定论**：之前文档里有"binwalk 在 macOS 兼容性问题"的定论作废——**binwalk CLI 完全可用，bug 在 adapter 层关键字白名单过窄**

## 1. 背景

### 1.1 当前 bug（`src/automisc/tools/shared/binwalk.py:21-38`）

```python
_FILE_HEADER_KEYWORDS = [
    "PNG image", "JPEG image", "GIF image", "PDF document",
    "ZIP archive", "RAR archive", "7-zip archive",
    "gzip compressed", "bzip2 compressed", "xz compressed",
    "tar archive", "ELF ", "PE32 ", "Microsoft Office",
    "OpenDocument", "pcap",
]
# 缺: "PEM private key" / "SSH private key" / "RSA private key"
```

binwalk 真扫到 `PEM private key` 两行（offset 0x51D60 + 0x52A7D），adapter 在 `binwalk.py:64-68` 用 `next((kw for kw in _FILE_HEADER_KEYWORDS if kw.lower() in desc.lower()), None)` 过滤，**没匹配上** → 整行丢掉 → 0 suspicious_point。

### 1.2 第二个 bug — adapter 不调 `binwalk -e` 提取

adapter 只跑 `binwalk <file>` 扫描，**不调 `binwalk -e` 提取**。Owner 在终端手动 `binwalk -e` 跑出 2 个 `pem.key` 在 `extractions/greatescape.pcap.extracted/51D60/pem.key`，但 GUI 用户没有这个能力。

### 1.3 Owner 提的关键 insight

> "理论上流量中包含的文件应该可以通过foremost提取吧？ 不论是ftp-data, 还是http又或者是其它协议。"

→ **adapter 应该走 magic bytes 路线（foremost / binwalk -e）通用解 pcap 里所有协议传输的文件**——不是按协议白名单手撕。这跟 pcap_protocol_router 的"协议路由"是**两个不同层**：
- pcap_protocol_router：**协议层**——告诉用户"哪个协议可能藏 key"
- binwalk adapter：**二进制层**——直接按 magic bytes 把文件雕出来

## 2. 设计决策（Owner Q&A）

### Q1: 哪些 magic 关键字要加到白名单？

**Owner 答**（B 方案默认走）：
- **加**：`"PEM private key"` / `"SSH private key"` / `"RSA private key"`（CTF / Forensics 流量题核心场景）
- **保留**：现有 PNG / JPEG / ZIP / 7z / ELF / PE 等 17 项
- **未来扩展位**：`"SSH public key"` / `"X.509 cert"` / `"PGP"`（v0.5+ 后续按真实题加）

### Q2: 提取后输出位置？

**Owner 答**：
- **同 input 目录**（per v0.5-output-samedir 规范）—— 跟 foremost 行为一致
- 路径格式：`<input_dir>/<input_stem>__binwalk_extracted/`
- 多个提取文件按 binwalk 默认子目录 `extractions/<input_stem>.extracted/<offset_hex>/<ext>`

### Q3: 提取命令参数？

**Owner 答**：
- 用 `binwalk -e <file>`（默认行为，所有 signature 都提取）
- **不**用 `binwalk -e --dd='.*'`（更激进但容易误伤）
- 跟 foremost 一样，写到 samedir；不污染 /tmp

### Q4: SuspiciousPoint 怎么显示提取结果？

**Owner 答**：
- 每个 magic 命中 1 个 SP（severity=4，因为还要用户确认）
- `context` 字段含**提取文件路径**（GUI 点击能跳）
- `suggested_action` 提"Wireshark 配 key 解密"（per pcap_protocol_router §4.3 模板）

## 3. 架构

### 3.1 修改点

```
src/automisc/tools/shared/binwalk.py
  - _FILE_HEADER_KEYWORDS: 加 3 项 (PEM private key / SSH private key / RSA private key)
  - 新增 _extract_files(file_path, outdir) 函数: 调 binwalk -e + 扫描 extraction 目录
  - run() 流程:
      1. 调 binwalk <file> 扫描 (原行为)
      2. 解析 magic hits → 过滤 _FILE_HEADER_KEYWORDS → SP list
      3. (新增) 调 binwalk -e <file> -o <samedir> 提取
      4. 扫描 extraction 目录 → 把提取文件路径填到 SP context
      5. 多个提取 → 第一个 SP 含全部路径列表
```

### 3.2 行为表

| binwalk 输出 | 现状 (bug) | 修复后 |
|---|---|---|
| `PEM private key @ 0x51D60` | 0 SP（关键字白名单不含）| severity=4 SP + `context: "extracted: <path>/51D60/pem.key"` |
| `ZIP archive @ 0x1234` | severity=4 SP（已工作）| 不变 + 提取到 samedir |
| `PNG image @ 0x5678` | severity=4 SP | 不变 + 提取 |
| `Windows shellcode @ 0xABCD` | 0 SP（关键字不含）| 0 SP（保持，非文件 magic）|
| 0 hits | 0 SP | 0 SP（不强制提取）|

### 3.3 decision tree

```
BinwalkAdapter.run(file_path)
  1. cmd1: `binwalk file_path` → 扫描
  2. 解析 → SP list (用扩展后的 _FILE_HEADER_KEYWORDS)
  3. cmd2: `binwalk -e file_path -o <samedir>` → 提取
  4. 扫描 <samedir>/<stem>__binwalk_extracted/extractions/<stem>.extracted/...
  5. 把提取文件路径填到 SP context
  6. 返回 ToolResult
```

## 4. 实现细节

### 4.1 关键字白名单扩展

```python
_FILE_HEADER_KEYWORDS = [
    # 现有 17 项
    ...
    # v0.5-binwalk-extract 新增 (per Owner Q1)
    "PEM private key",   # RSA / EC / generic PEM (OpenSSL etc.)
    "SSH private key",   # OpenSSH 私钥
    "RSA private key",   # 旧版 RSA
]
```

### 4.2 提取函数（伪代码）

```python
def _extract_files(self, file_path: str, outdir: Path) -> list[Path]:
    """调 binwalk -e 提取嵌入文件到 outdir。"""
    # 清理旧输出（binwalk 会因目录非空而 exit 1）
    if outdir.exists():
        shutil.rmtree(outdir)
    outdir.mkdir(parents=True, exist_ok=True)

    cmd = [
        self.binary_path or "binwalk",
        "-e",         # extract
        file_path,
        "-o", str(outdir.parent),  # binwalk -o 是父目录, 自己加 _extracted 子目录
        "--run-as=root",  # macOS 某些情况需要
    ]
    self._run_subprocess(cmd)

    # binwalk 默认子目录: <outdir.parent>/extractions/<stem>.extracted/
    extracted_root = outdir.parent / "extractions" / Path(file_path).name + ".extracted"
    if not extracted_root.exists():
        return []
    return sorted(extracted_root.rglob("*"))  # 包含子目录
```

### 4.3 SP 模板

```python
SuspiciousPoint(
    id="",
    tool_name="binwalk",
    file_path=pcap_path,
    category="file_header",
    offset=offset,
    matched_pattern=f"{matched_kw} @ offset {offset}",
    severity=4,
    suggested_action=(
        f"已提取到 {outdir}/extractions/{stem}.extracted/{offset_hex}/{ext}\n"
        f"如需用此文件解密 TLS: 配 Wireshark --ssl.keys <server_ip>,<port>,http,<key_path>"
    ),
    context=(
        f"extracted_files={','.join(str(p.relative_to(outdir)) for p in extracted)}"
    ),
)
```

## 7. 风险 & 缓解

| 风险 | 缓解 |
|---|---|
| binwalk -e 提取所有 signature，可能误伤（提取垃圾文件）| 默认 `binwalk -e` 已很保守；不调 `--dd` 更激进选项 |
| 提取目录体积大 | 写到 samedir，Owner 可手动删 |
| macOS `--run-as=root` 报错 | 不加这 flag；binwalk 不需要 root（除非读 raw device，本迭代不处理）|
| SuspiciousPoint context 字段过长（多个 extracted 文件）| 限 5 个文件，截断 |
