## 1. 背景

### 1.1 问题陈述 (per v0.5-train-009 §2-§3)

N=NP (多答案) 这道题:
- 图: 85×471 PNG,8999 bytes
- 数据藏在哪: G 通道 LSB(bit 0)**列扫描**字节流 → unhexlify → pyc 文件 → 执行 → flag
- writeup 写法: PIL `getpixel((w,h))[1] & 1` 逐像素取 LSB,**外层 w 内层 h = 列扫描**

**automisc 当前能力 gap**:

| 维度 | 现状 (v0.5-LSB-router) | writeup 需求 | 缺口 |
|---|---|---|---|
| 抽取通道 | zsteg 固定的 `b1,*,lsb,xy` LSB 通道组合 | G 通道 bit 0 + 列扫描 | ❌ 写死 |
| 扫描顺序 | zsteg 默认行扫描 | 外层 w(列)内层 h(行) = 列扫描 | ❌ 不可改 |
| 字节内 bit 序 | MSB first | MSB first(碰巧一致) | ⚠️ 应可配置 |
| 抽到 raw bytes 怎么用 | 直接走 `detect_magic(data[:16])`(只扫 offset 0) | unhexlify → 写 pyc → 执行 | ❌ 不可控 |

### 1.2 现有架构不动 (per v0.5-train-009 §4.2 Owner 决策)

**保留 `LSBExtractAction` (基于 zsteg)** —— 默认路径不变,GUI 用户拖文件仍然走 zsteg 推荐,**新入口跟现有入口并行**。

## 2. 目标

新增 2 个独立能力,跟现有 LSBExtractAction 并行:

### 2.1 能力 B: `LSBBytesExtractAction`

**目标**: 从 PNG/BMP/GIF(任意像素格式)任意通道 × 任意 bit 位 × 任意扫描顺序 × 任意字节内 bit 序,抽出 raw bytes,写到 input 同目录。

### 2.2 能力 C: `magic_sniffer` decoder

**目标**: 对任意 raw bytes(从能力 B 抽出,或外部拖入),滑动窗口匹配 20+ 文件 magic,**不只看 offset 0**,命中 → severity=5 + 写 `<stem>__sniffed.<ext>` + journal 高亮。

### 2.3 chain `lsb-bytes` 串联

**目标**: 新 chain `lsb-bytes` = `binwalk_extract → LSBBytesExtractAction → magic_sniffer`,一次跑完 N=NP 类题全流程。

### 3.5 magic_sniffer 嗅探算法

```python
def sniff_magic(data: bytes, max_offset: int = 32) -> list[SniffResult]:
    """滑动窗口扫 offset 0..max_offset, 字节对齐 1/2/4.
    
    Returns:
        [(offset, magic_desc, ext, severity)] 按 offset 升序
    """
    results = []
    for offset in range(max_offset + 1):
        for sig, desc in MAGIC_SIGNATURES:
            if data[offset:offset + len(sig)] == sig:
                ext = _ext_from_magic_desc(desc)
                results.append((offset, desc, ext, 5))
    return results
```

**关键点**:
- 滑动窗口扫 offset 0~32(覆盖"前缀数据 + magic + 后缀"场景)
- **不只看 offset 0**(per `router.detect_magic` 痛点)
- 命中后 severity 统一 = 5(可疑,需要人工/auto-run 二次处理)
- 字节对齐 1/2/4(Mach-O 64-bit、ELF 等是 4 字节对齐 magic)

## 7. 待 Owner 拍板的决策点

### Q1: `lsb-bytes` chain 默认参数?

- **A**: 全跑 3 通道 × bit 0/7 × row/col × MSB = 12 组合,慢但覆盖广(预计 ~5s/张 PNG)
- **B**: 默认只跑 G 通道 bit 0 row MSB(快,~0.5s,但漏掉 writeup 这类 G 通道 col 场景)
- **C**: 默认跑 3 通道 × bit 0/7 × row/col = 12 组合(等价 A,措辞改)

**Mavis 建议**:A/C — 跑全 12 组合,反正都是 numpy 切片,< 5s,覆盖最广。**默认 12 组合,user 想精确控制可以 CLI/GUI 指定**。

### Q2: `magic_sniffer` 嗅探命中后是否自动执行文件?

- **A**: 只写 sniffed 文件 + journal 高亮 + flag_candidate,**不自动执行**(半自动化辅助,Owner 决策下一步)
- **B**: 命中 .pyc / .exe 直接执行 / 命中 .zip 直接解压(全自动,但有安全风险)

**Mavis 建议**:A — automisc 是半自动化辅助(per STRUCTURE.md §1),不抢 flag。**写文件 + 高亮 + 让 Owner 决定**。

### Q3: GUI 参数 dialog 是 modal 还是 inline?

- **A**: Modal dialog(`QDialog` + 4 dropdown + OK/Cancel),简单直接
- **B**: Inline panel(右侧永久显示 4 dropdown + "Run" 按钮),所见即所得

**Mavis 建议**:A — modal dialog 跟现有 "Chain Runner" 风格一致,改动小。

### Q4: chain 输出 .bin 文件要不要默认清理?

- **A**: 默认保留(Owner 二次分析用),CLI `--keep-bin` flag 强制保留
- **B**: 默认清理(只留 sniffed 命中文件),CLI `--keep-bin` flag 强制保留

**Mavis 建议**:A — 默认保留,跟 v0.5-output-samedir "持续输出都写到 input 同目录" 治理一致(per Owner 06-14 拍板)。

### Q5: `purpose="lsb-bytes"` 文件名格式是否带全部 4 个参数?

- **A**: `np__lsb_g_b0_col_msb.bin`(全部 4 参数)
- **B**: `np__lsb_g.bin`(只带 channel,其他默认参数不重复)

**Mavis 建议**:A — 完整记录,Owner 一眼看出"用了哪个组合",reproduce 时不用查 chain log。

### Q6: 实施后是否需要 `regression 测试现有 Challenge/steg.png`?

## 8. 风险 + 缓解

| 风险 | 缓解 |
|---|---|
| `LSBBytesExtractAction` 对非 PNG 格式图片(BMP/GIF)行为不一致 | v0.1 起步只支持 PNG + BMP,BMP 后续按需扩;Q1 默认参数先跑 PNG |
| `magic_sniffer` 误报(magic 巧合命中) | severity=5 标红 + 不自动执行,Owner 二次确认 |
| `binwalk_extract` 链头对 PNG 卡死(参考 train-003 教训) | 链头加 timeout,跑前先 file 检查 magic |
| GUI 参数 dialog 阻塞主线程 | 用 `QDialog.exec()` modal,不阻塞主线程(只阻塞当前 dialog) |

| 跟现有 `lsb` chain 命名混淆 | CLI `chain --chain lsb-bytes` 显式区分;GUI 菜单 "lsb-bytes" 单独入口 |
