## 1. 背景

### 1.1 现状

`src/automisc/core/actions/zip_chain.py` 的伪加密算法是**粗粒度**实现：

| 组件 | 现状 | 问题 |
|---|---|---|
| `_is_pseudo_encrypted(zip_path) -> bool` | 任一 entry 是伪加密 → True | **不知道**哪些 entry 是伪加密 |
| `FixPseudoEncryptionAction.run` | 清**所有**外层 LFH/CDH flag bit 0 | **不区分**伪加密 vs 真加密 entry |
| `TryUnzipAction.run` | 整体 fail → is_pseudo | fail 时**只**返回 bool，调用方不知详情 |

### 1.2 owner 反馈核心

> "00000038.zip 有一部分数据是伪加密，一部分是真加密。你应该按照 ctfwiki 的中 zip 伪加密的原理进行算法调整，不要认为全部数据都是伪加密，也不要因为第一个数据区是真加密就停止对后面的数据区的伪加密判断。"

**owner 真实诉求**：
1. **逐 entry 独立判断**（per-entry classification）
2. **不 short-circuit**（per entry）— 遇到第一个真加密 entry **不**停
3. 修：**只**清**伪加密** entry 的加密位，**不**动**真加密** entry

## 2. 设计

### 2.1 新增 `_classify_zip_entries` 函数

**接口**：
```python
def _classify_zip_entries(zip_path: Path) -> dict:
    """逐 entry 分类 (per ctf-wiki 原理 + v0.5-train-005 反馈).
    
    Returns:
        {
            "pseudo": {fname: (lfh_offset, cdh_offset)},   # 伪加密
            "real":   {fname: (lfh_offset, cdh_offset)},   # 真加密
            "clear":  {fname: (lfh_offset, cdh_offset)},   # 完全明文
        }
    
    算法 (per ctf-wiki 原理):
    1) 收 LFH map (fname → (lfh_offset, flag, comp_size, data_start))
    2) 收 CDH map (fname → (cdh_offset, flag)) — 用 EOCD 倒推外层 CDH 区域
    3) 逐 entry 判断:
       - LFH bit0=0 AND CDH bit0=0 → clear (不需要修)
       - LFH bit0=1 OR CDH bit0=1:
         - comp_size < 12 → pseudo (太短不可能有真加密 PKCS#5 header)
         - data[data_start + 11] in range(12) → real (有 PKCS#5 末位)
         - else → pseudo (无 PKCS#5 末位)
    4) 不 short-circuit — 扫完所有 entry
    """
```

### 2.2 `_is_pseudo_encrypted` 升级（保持向后兼容）

```python
def _is_pseudo_encrypted(zip_path: Path) -> bool:
    """检测 zip 是否含伪加密 entry.
    
    升级 (per v0.5-train-005): 内部用 _classify_zip_entries,
    返回 len(classify['pseudo']) > 0.
    
    向后兼容: 返回 bool, 现有调用方不需改.
    """
    try:
        classify = _classify_zip_entries(zip_path)
    except Exception:
        return False
    return len(classify["pseudo"]) > 0
```

### 2.4 per-entry 分类边界规则 (per v0.5-train-006 修复)

**v0.5-train-006 修复**: 弃用 `data[data_start + 11] in range(12)` 启发式 (命中率仅 4.7%), 改用 **zlib.decompress + CRC-32 校验** 作为客观判据.

| LFH bit0 | CDH bit0 | comp_size | method | data 解压 + CRC | 分类 |
|---|---|---|---|---|---|
| 0 | 0 | * | * | n/a | **clear** (不需要修) |
| * | * | < 12 | * | n/a | **pseudo** (太短不可能有真加密 PKCS#5 12-byte header) |
| * | * | >= 12 | 0/8 | 直接 inflate/stored CRC 匹配 | **pseudo** (data 是明文, flag 改了) |
| * | * | >= 13 | 8 | 跳过 12-byte header 后 inflate + CRC 匹配 | **real** (data 是加密 deflate, header 后是密文) |
| * | * | >= 12 | * | 都不匹配 | **real** (保守归类, per §4「保守归类原则」) |

**新算法核心**:
```python
# 1) 直接 inflate (无 PKCS#5 header) → 伪加密特征
if _check_pseudo(file_data, crc, method):
    pseudo[fname] = ...
    continue

# 2) 跳过 12-byte PKCS#5 header 再 inflate → 真加密特征
if comp_size > 12 and _try_inflate_with_crc(file_data[12:], crc):
    real[fname] = ...
    continue

# 3) 都不通过 → 保守归 real
real[fname] = ...
```

**为什么新算法对**:
- **不**依赖 byte 位置启发式, 改用 **ZIP 标准自带的 CRC-32** 作判据
- **不**需要密码就能判断 (CRC 是 header 里的明文字段, 不依赖解 data)
- 命中率高: CRC-32 碰撞概率 1/2^32, 几乎不会误判
- 同时覆盖 deflate (method 8) 和 stored (method 0) 两种压缩方式

**保守归类原则** (per v0.5-train-006 §4):
- 都不能 deflate 时, 默认归 **real** (不归 pseudo)
- 理由: pseudo → user 跑 ZipCenOp r (破坏性, 改字节后**不能**验证是否真解出)
       real → user 跑 bruteforce (非破坏性, 跑 1 分钟没结果就走密码学/手工)
- **误判 real 比 误判 pseudo 安全**: real 让用户保留文件, pseudo 误导用户做破坏性修改
