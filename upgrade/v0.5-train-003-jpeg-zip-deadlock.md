## 2. automisc 行为对比（auto-run 卡住根因）

### 2.1 跑过的工具（exiftool / binwalk / strings）

| 工具 | 结果 | 时间 | 命中 |
|---|---|---|---|
| exiftool | exit 0, 1 SP severity=3 (Comment metadata) | 0.19s | "gd-jpeg v1.0 / quality=90" |
| binwalk | exit 0, 2 SP severity=4 | 0.06s | "JPEG image @ 0" + **"ZIP archive @ 68019"** |
| strings | exit 0, 5 SP severity=5 | 0.07s | "flag.txtUT" (2 次) + Base64 串 |

**3 个工具总时间 0.32s**——auto-run 本身没卡。

### 2.2 卡住根因（gui/main_window.py:223-336）

`_on_auto_chain_finished` 触发 `_maybe_trigger_zip_chain_from_binwalk`，**关键路径**：

```
exiftool OK
binwalk OK → 命中 "ZIP archive @ 68019"
strings OK (severity=5 终止 short-circuit)

→ _on_auto_chain_finished 调 _maybe_trigger_zip_chain_from_binwalk
→ 调 BinwalkExtractAction（foremost 雕 zip, 0.04s, 成功提取 /tmp/.../zip/00000132.zip）
→ 对每个 .zip 跑 build_zip_chain_with_bruteforce() (zip-full chain)
→ dag.execute(...) 在 **GUI 主线程同步调**
→ BruteforceZipAction 调纯 Python zipfile 试 8.4M 密码
```

**问题 1（主线程死锁）**：`dag.execute()` 在 GUI 主线程同步跑，8.4M 密码字典 100 密码 1.6s，**预估 37 小时**（per 实测 100 密码耗时 1.6s × 84000 倍）。

**问题 2（chain 选择过激）**：binwalk 命中 archive → 立刻 `zip-full`（含 bruteforce），但 **binwalk 报"ZIP archive" 可能是加密的（密码未知），也可能是伪加密的（密码空）**——两种情况处理方式完全不同。当前代码一刀切全跑 bruteforce。

**问题 3（无超时）**：`BruteforceZipAction` 默认 120s 超时，但 dag.execute 是嵌套调用，超时机制不传递。

### 2.3 完整模拟（无 GUI）

```python
$ python -c "
from automisc.core.actions.zip_chain import BruteforceZipAction
import time
t0 = time.time()
BruteforceZipAction().run({
    'file_path': '/tmp/zip_einstein/extracted.zip',
    '__bruteforce_limit__': 100,  # 限 100 个
})
print(f'100 密码: {time.time()-t0:.2f}s')
"
# 输出: 100 密码: 1.60s
# 8.4M 密码预估: 1.6 × 84000 / 60 ≈ 2240 分钟 ≈ 37 小时
```

**实测 100 密码 1.6s → 8.4M 密码 37 小时**（永远跑不完）。

## 3. automisc 应补能力（架构升级输入）

| 能力 | 现状 | 应补 |
|---|---|---|
| **`dag.execute` 走 GUI 主线程** | 同步阻塞主线程 | **改走 `ChainRunner` (QThread 异步)** —— `auto-run` 检测到 archive → emit signal → GUI 弹"检测到 zip, 是否跑 chain"对话框 |
| **`build_zip_chain_with_bruteforce` 触条件** | binwalk 报 archive → 立刻跑 | 加 `__bruteforce_limit__` 默认值 (8.4M 字典太慢) + **先 try_unzip（密码空/伪加密直接解）→ 失败再 bruteforce** |
| **BruteforceZipAction 超时** | 默认 120s 但 dag.execute 不传递 | chain 级别 timeout 强制（5 分钟硬切）→ 失败转"密码未知, Owner 手工处理"|
| **主线程 progress 显示** | 完全卡住 | brutalforce chain 跑时 status bar 显示 "bruteforce 8.4M / 当前 0.1M" 增量进度 |
| **chain 选择 smart-routing** | binwalk 一刀切走 zip-full | 加启发式: zip 文件大小 / flag bit 0 / data_descriptor 标志 → 走 `zip` (无 bf) 或 `zip-full` |

## 4. 架构判定（per §5.2）

| 判定项 | 结论 |
|---|---|
| 同类样本 | **≥ 3 类**：（a）JPEG 嵌加密 zip（这题）；（b）PNG 嵌加密 rar；（c）PDF 嵌加密 7z；（d）任何 archive 容器 |
| 架构层 vs 单题 | **架构层**——auto-run 主线程同步调 dag.execute + 8.4M 字典 brutalforce 永远跑不完是**跨工具 / 跨 archive 类型**的 UX 缺陷 |
| 触发架构升级 | ✅ **是** → 触发 `upgrade/v0.5-auto-run-async-chain.md`（待建）|
| 训练日志直接改代码 | ❌ **否**（per §5.5）——只描述问题，架构升级走单独迭代 |
