## 1. 背景

### 1.1 问题陈述 (per Owner 06-21 12:40 反馈)

Owner 实测发现:
- 拖图片 auto-run 跑 [zsteg / stegseek / exiftool / binwalk / strings / file] 6 个工具
- zsteg 在 N=NP 类 LSB 题**漏报**(默认行扫描 + 写死通道)
- `lsb_bytes_extract` **没接进 auto-run 池** (要 GUI 工具栏或 CLI 显式跑)

Owner 反思: 既然 auto-run 的 lsb (zsteg) 已经"判断 + 抽 + output",`lsb_bytes_extract` 是不是应该一样接进去?

### 1.2 根因 (per v0.5-lsb-byte-stream-extract spec §3.2)

v0.5-lsb-byte-stream-extract (commit `a528f65`) 设计时:
- ✅ CLI 完整支持
- ✅ 新 chain `lsb-bytes` 注册到 `core.chains.build_lsb_bytes_chain`
- ❌ **`LSBBytesExtractAction` 是 action 不是 adapter** (走 `core.actions` 不走 `core.tools`)
- ❌ **`core.run_tool` 只接 adapter** (从 `get_tool_class(tool_name)` 拿)
- ❌ **`FIND_SUSPICIOUS_PICTURE_TOOLS` 没加 lsb_bytes_extract**

设计意图是 user-controlled 4 参数(per v0.5-lsb-byte-stream-extract spec §1.2 "保留 LSBExtractAction, 现有架构不动"),但**没考虑 auto-run 池自动跑默认参数**。

### 1.3 Owner 决策 (按 12:43 "按你的建议")

> "先做 v0.5-lsb-bytes-gui (GUI 入口 + dialog), auto-run 留第二轮迭代"

第一轮 `v0.5-lsb-bytes-gui` (main `c898a46`) 已落地 GUI user-controlled 入口。本迭代是**第二轮** — 把 `lsb_bytes_extract` 接进 auto-run 兜底。

### 1.4 与 v0.5-lsb-bytes-gui 的互补关系

| 路径 | 触发 | 默认行为 | 适用场景 | 迭代 |

## 2. 目标

### 2.1 能力 F: auto-run 池加 lsb_bytes_extract 兜底

**目标**: 拖图片后 auto-run 自动跑 lsb_bytes_extract(默认 12 组合),让 zsteg 漏的 N=NP 类 LSB 题也能命中。

### 2.2 关键约束

| 约束 | 详情 |
|---|---|
| **走 core.run_tool 统一入口** | 不能改 `find_suspicious_from_picture` 接口, 因为其他 4 pool (picture/traffic/archive/binary) 都走 `core.run_tool` |
| **不动 LSBBytesExtractAction** | action 在 `core/actions/`, 跟 adapter 解耦设计 |
| **不动现有 GUI 入口** | GUI user-controlled (v0.5-lsb-bytes-gui) 保持 |
| **不抢 magic_sniffer** | magic_sniffer 是 decoder 不混进 Action 链 (per v0.5-lsb-byte-stream-extract spec §3.1) |

### 3.2 渲染流程 (跨文件追踪)

```
1. GUI 拖 PNG → main_window.dropEvent
2. main_window → find_suspicious_runner.start(file_path)
3. FindSuspiciousRunner.run → pick_suspicious_pool(file_path) = ("picture", [..., "lsb_bytes_extract"])
4. for tool_name in pool:
     core.run_tool("lsb_bytes_extract", file_path)
     → LsbBytesExtractAdapter().run(file_path)
       → LSBBytesExtractAction.run(context={"file_path": file_path})  # 默认参数
       → 抽 12 个 .bin 文件 → 写到 input 同目录
       → ActionResult(success=True, data={"extracted_files": [12 paths]})
     → 转 ToolResult(success=True, suspicious_points=[SP(severity=5, "lsb_bytes_extracted")])
     → journal.record(...)
5. auto-run 跑完后, GUI 主线程渲染 SP 到 journal_panel
6. Owner 看 SP: severity=5 "lsb_bytes_extracted" → 知道 LSB 字节流抽到内容
```

### 3.3 关键设计决策

#### 3.3.1 为什么新建 adapter 而不是改 LSBBytesExtractAction

- `core.run_tool` 接口固定 `(tool_name, file_path)`,没法传 4 参数
- LSBBytesExtractAction 默认参数已经支持 96 组合的子集(12 组合 = 实战常用)
- adapter 包一层 = 走 run_tool 统一入口, 不破坏 action 抽象

#### 3.3.2 默认 12 组合 vs 24 组合

| 组合数 | channels | bit | scan_order | byte_bit_order | 跑时 | 覆盖度 |
|---|---|---|---|---|---|---|
| **12 组合 (A)** | RGB (3) | 0/7 (2) | row/col (2) | MSB (1) | ~5s | N=NP 题 G 通道 col 命中 |
| 24 组合 (B) | RGB (3) | 0/7 (2) | row/col (2) | MSB/LSB (2) | ~10s | 同 A + LSB-first 场景 |
| 6 组合 (C) | RGB (3) | 0 (1) | row/col (2) | MSB (1) | ~2.5s | 漏 bit 7 (高位 LSB), 实战偶尔需要 |

**Mavis 建议 A** (per "可疑点越多越好" 铁律 + v0.5-lsb-byte-stream-extract spec §3.3 默认就是 RGB × bit 0 × row × MSB 的扩展)。

#### 3.3.3 抽出 .bin 文件保留 vs 清理

| 选项 | 优点 | 缺点 |
|---|---|---|
| **保留 (A)** | 跟 v0.5-output-samedir 治理一致 (per Owner 06-14 拍板), Owner 二次分析用 | 12 个 .bin 占磁盘 |
| 清理 (B) | 省磁盘 | Owner 没法看 raw 字节流 |

**Mavis 建议 A** — auto-run 跟 GUI user-controlled 一致保留。

#### 3.3.4 跑完后接不接 magic_sniffer 自动 sniff

| 选项 | 行为 | 理由 |
|---|---|---|
| **不接 (A)** | adapter 只写 .bin + SP, 不接 magic_sniffer | magic_sniffer 是 decoder 不混进 Action 链 (per v0.5-lsb-byte-stream-extract §3.1 决策) |
| 自动 sniff (B) | adapter 写完 .bin 后调 magic_sniffer 嗅探 | 违反架构分层, 而且 sniff 命中后还要执行 (auto-run 不抢执行) |

**Mavis 建议 A** — 不接。Owner 跑完 auto-run 后, 看 SP 自己决定跑不跑 magic_sniffer。

#### 3.3.5 异常处理 (per v0.5-auto-run-discipline)

- **PIL/numpy 没装**: 返 SP severity=2 "lsb_bytes_extract: PIL/numpy 缺失", 不中断其他工具
- **JPEG/GIF 格式不支持某些通道**: 跳过该组合, 不中断
- **文件不存在 / 损坏**: 返 SP severity=2 "lsb_bytes_extract: file error", 不中断
- **超大图 (>100MB)**: 走 default_timeout=30s, 超时返 SP severity=2
