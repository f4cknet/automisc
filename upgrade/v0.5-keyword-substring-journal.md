### 1.2 根因分析

`src/automisc/core/suspicious.py` 的 `_keyword_pattern`:
```python
def _keyword_pattern() -> re.Pattern:
    escaped = [re.escape(k) for k in KEYWORDS]
    return re.compile(r"\b(?:" + "|".join(escaped) + r")\b", re.IGNORECASE)
```

🚨 **3 个 bug 复合**:

| Bug | 现象 | 影响 |
|---|---|---|
| 1. `\b...\b` word boundary | `this_is_not_password` 里的 `password` 前面是 `_`, word boundary 失效 | owner 真实样本 exiftool 漏匹配, journal 不收 |
| 2. KEYWORDS 缺 `ctf` / `pass` / `f1ag` / `p@ssw0rd` | 8 个 owner 拍板的关键字只含 1 个 (password) | 大部分高优先级 keyword 漏匹配 |
| 3. SEVERITY_MAP["keyword"] = 1 | 不一致: rule_scanner 5 vs suspicious 1 | journal 显示不红, 等级低 |

**为什么 GUI 高亮对了?**
- GUI 渲染层 (`gui/output_view.py`) 走 `_SENSITIVE_KEYWORDS` (rule_scanner.py:39) — 子串匹配
- Journal 累积走 `suspicious.py:_keyword_pattern` — `\b` 边界
- 两层**用不同的匹配规则**, 是设计上的重大缺陷 (per memory 教训 "GUI 渲染层和核心层解耦没做好")

```python
    "jsteg", "zsteg", "binwalk", "foremost",

```

#### B. `core/suspicious.py:_keyword_pattern` — 子串匹配 + 长 keyword 优先

```python
def _keyword_pattern() -> re.Pattern:
    # 长度降序 — `password` (8) 在 `pass` (4) 之前,
    # 否则正则 alternatives 按顺序匹配会先吃 `pass`,
    # `password=hunter2` 命中 `pass` 而不是 `password`
    escaped = sorted((re.escape(k) for k in KEYWORDS), key=len, reverse=True)
    return re.compile("(?:" + "|".join(escaped) + ")", re.IGNORECASE)
```

#### C. `core/suspicious.py:SEVERITY_MAP` — keyword severity 1 → 5

```python
SEVERITY_MAP: dict[str, tuple[int, str]] = {
    "flag": (5, "直接拿 flag 提交"),
    # per Owner 18:03 铁律: 高优先级 keyword 命中 = severity 5
    # 不触发 short-circuit — SHORT_CIRCUIT_SEVERITY=99 (per v0.5-journal-highlight-keywords Q12)
    "keyword": (5, "高优先级可疑关键词命中 ..."),
    ...
}
```

#### D. `gui/journal_panel.py:__init__` — 标题改

```python
def __init__(self, parent=None) -> None:
    super().__init__("可疑点列表", parent)  # per Owner 18:03
```

### 3.2 为什么删 `\b` word boundary

- `\b` 是 word boundary, 在 `_`, `-`, `0`, `@` 等字符处失效
- 实战样本: `this_is_not_password`, `passphrase`, `p@ssphrase`, `the_password_field`, `pass_word` 全部命中失败
- 子串匹配 (`if kw in text_lower`) 才是 CTF 实战真正需要的

### 3.3 为什么 length 降序

re 默认按 alternatives 顺序匹配, **第一个匹配成功**就返回:
- 输入 `password=hunter2`
- KEYWORDS 含 `pass` (4) 和 `password` (8)
- 升序: `pass|password` → 先匹配 `pass` → 截短成 `pass` ❌
- 降序: `password|pass` → 先匹配 `password` (8 字符,贪婪) → 命中 `password` ✅

### 3.4 为什么 severity=5

- owner 铁律明确: "命中 → 加进 GUI '可疑点列表'" (重要信号)
- 与 `rule_scanner.CATEGORY_SENSITIVE` severity=5 保持一致 (统一 layer 行为)
- 不触发 short-circuit — `SHORT_CIRCUIT_SEVERITY=99` (per v0.5-journal-highlight-keywords Q12 拍板)
