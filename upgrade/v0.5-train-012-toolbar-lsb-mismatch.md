## 3. AI 兜底 (per §5.3 step 3, 验证 bug 根因 + 设计修复)

### 3.1 bug 根因 (per code review)

**lsb_extract.py 抽字节流** (line 206):
```python
# lsb_extract.py text 分支:
raw = _run_zsteg_extract(file_path, entry["channel"])  # 调 zsteg -e
```
- 调 `zsteg -e <channel>` 抽 zsteg stdout 命中的 channel 字节流
- zsteg stdout 优先 text 通道 (per `_TEXT_CHANNEL_PRIORITY` line 45-56)
- text 通道抽 printable text 字节流 (e.g. "Hey I think..." 117 byte)

**lsb_detect.py 抽字节流** (line 342-365):
```python
# lsb_detect.py 12 组合 LSB (per v0.5-train-011 修复):
bit0_planes = arr[:, :, list(perm)] & 1
if scan == "row":
    bits_flat = bit0_planes.flatten()  # HWC interleaved
# 8 bit 拼 byte, MSB first
```
- 用 PIL/numpy HWC interleaved 抽 12 组合 (6 排列 × 2 scan)
- 字节流内容跟 zsteg -e 抽的**完全无关**
- 命中 file_header 的 SP 对应 PNG 字节流, zsteg stdout 没命中 (因为 zsteg stdout 只看 printable text 段, 跳过 file_header 段)

**两条路径字节流不同源 = bug 根因**

### 3.2 Owner 实战预期

- 工具栏"PNG LSB 智能提取" 应该写真 **lsb_detect 命中的 SP** 对应字节流
- 不是 zsteg stdout 命中的 text channel
- 写真 `flag11__lsb.png` (PNG magic 89 50 4E 47 命中 → .png 后缀), 26934 byte, file 命令识别 `PNG image data`
- Owner 拿到 .png 后用 foremost / binwalk 二次分析

### 3.3 修复方向 (per spec v0.5-lsb-extract-pil-same-source, 待写)

**方案 A** (推荐, 最小变更):
- 改 `lsb_extract.py` 抽字节流逻辑, 用 PIL/numpy HWC (跟 lsb_detect 同源) **替代** zsteg -e
- 保留 zsteg 作为**兜底** (zsteg stdout 命中 text 时仍用 zsteg -e)
- 写真 magic 判定后缀 + write_bytes (per v0.5-lsb-extract-output-bytes 已有)
- 公共函数 `_extract_lsb_byte_stream(arr, perm, scan) -> bytes` 抽 LSBDetectAction + LSBExtractAction 复用

**方案 B** (废弃 zsteg, 全面 PIL/numpy):
- 完全废弃 zsteg 依赖, 12 组合全 PIL/numpy
- 不再走 zsteg -e 路径
- 改动面大, lsb_extract 整体重写

**方案 C** (新工具, 工具栏入口重定向):
- 新工具 `lsb_extract_from_sp` 复现 lsb_detect 命中的 SP 字节流
- 工具栏"PNG LSB 智能提取" 改成调新工具
- 老的 lsb_extract (zsteg) 保留供未来手工调用
- 改动面小, 但引入新工具

**Mavis 建议 A** (跟 v0.5-lsb-extract-output-bytes 修复一脉相承, 最小变更 + 同源 + 兜底):
