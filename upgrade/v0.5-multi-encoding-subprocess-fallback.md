## 1. 背景

### 1.1 Owner 真实样本 (2026-06-20 14:46)

Owner 实测 123456cry.jpg 完整流程时, GUI 显示 stegseek 抓 ko.txt 内容:

```
⬛⬛⬛⬛⬛⬛⬛⬛eT⬛⬛⬛⬛⬛⬛⬛oy⬛⬛⬛⬛⬛⬛ 蛋
```

`⬛⬛⬛` 是 U+FFFD 替代字符。但字节实际是 GBK 编码的"看到这个图片就是压缩包的密码:bV1g6t5wZDJif^J7"。

### 1.2 根因分析

`tools/base.py` 的 `_run_subprocess` 历史上两种状态:

**v0.1 时代 (commit 前)**:
```python
proc = subprocess.run(cmd, capture_output=True, text=True)
# text=True 默认 errors='strict'
# 跑 foremost / unzip 雕 binary 文件 → UnicodeDecodeError 挂掉
```

**commit d500d79 (Owner 12:50 反馈)**:
```python
proc = subprocess.run(cmd, capture_output=True, text=True, errors='replace')
# 不 crash 了, 但 GBK 中文 → U+FFFD ⬛⬛⬛
```

中文乱码的本质: Python 3.13 subprocess.run `text=True` 默认用 `locale.getpreferredencoding()` (= `utf-8`),对 GBK 输出直接 decode 失败,`replace` 模式把无效字节全替成 U+FFFD。

### 1.3 CTF 实战需求

CTF 中文题目极常见 (尤其是中国 CTF 比赛),owner 实战样本:

| 来源 | 编码 | 出现频率 |
|---|---|---|
| stegseek 抓 ko.txt | GBK | ⭐⭐⭐⭐⭐ (CTF 中文赛题标配) |
| zip 解压中文 flag 文件名 | GBK | ⭐⭐⭐⭐ |
| binwalk 雕 zip 注释 | GBK | ⭐⭐⭐ |
| exiftool 雕 jpg 注释 | UTF-8 / GBK 混合 | ⭐⭐⭐ |
| strings 抓 Windows PE 资源 | GBK | ⭐⭐ |

不修 = 中文 flag / 密码提示 / 备注全在 GUI 显示成 `⬛⬛⬛`,等于直接丢信号。

## 2. 目标

**让所有走 `_run_subprocess` 的工具输出,中文/日文/繁体中文都能正确显示,不再 ⬛⬛⬛ 替代。**

非目标:
- 不追求完美解码所有 edge case (latin-1 兜底 + 0 U+FFFD 选最优 = 足够)
- 不引入第三方编码检测库 (chardet 等, 保持 zero dependency)

## 3. 设计

### 3.1 核心改动: `_decode_output_bytes(bytes) -> str`

```python
_DECODE_CANDIDATES = ("utf-8", "gbk", "gb18030", "big5", "shift_jis", "latin-1")

def _decode_output_bytes(data: bytes) -> str:
    """多编码 fallback, 选 0 U+FFFD 的."""
    if not data:
        return ""

    # 阶段 1: strict 解码, 找 0 U+FFFD 的 (完美解码)
    for enc in _DECODE_CANDIDATES:
        try:
            return data.decode(enc)
        except UnicodeDecodeError:
            continue

    # 阶段 2: 全失败 → errors='replace' 选 U+FFFD 最少
    best_text, best_replacement = "", float("inf")
    for enc in _DECODE_CANDIDATES:
        text = data.decode(enc, errors="replace")
        replacement = text.count("\ufffd")
        if replacement < best_replacement:
            best_replacement, best_text = replacement, text
            if replacement == 0:
                break
    return best_text
```

### 3.2 为什么不用 chardet / cchardet

- 多 1 个 C 扩展依赖, 编译链复杂 (macOS Linux Windows 都得过)
- chardet 对短文本 (< 100 bytes) 准确率低 (stegseek 抓 ko.txt 48 字节典型)
- 候选集 `utf-8/gbk/gb18030/big5/shift_jis/latin-1` 已经覆盖 CTF 99% 场景
- strict first → 完美解码 0 U+FFFD = 最优解

### 3.3 为什么不用 `errors='ignore'`

- 静默丢字节 → foremost 解出 123456cry.jpg 但日志看不到关键片段 (per Owner 12:50 反馈)
- U+FFFD 比空字符串更直观, 用户能看到"这有乱码"而不是"这什么都没"

### 3.4 latin-1 为什么放最后兜底

- latin-1 (`ISO-8859-1`) 永远成功 (`bytes[0] → chr(bytes[0])` 一一映射 0x00-0xff)
- 但 latin-1 解读 GBK 双字节字符 = 显示 `\xa5\x68` 为 `\xa5\x68` 两个独立字符 (不是合并成中文字)
- 放最后 = 阶段 1 strict 解码走不到 latin-1 (utf-8/gbk 等先成功), 真 random binary 才 fall through
- 阶段 2 errors='replace' 兜底: latin-1 永远 0 U+FFFD, 所以 random binary 走 latin-1 也 OK

### 3.5 为什么 `text=False` 模式

```python
# 之前 (commit d500d79)
proc = subprocess.run(cmd, capture_output=True, text=True, errors="replace")
# stdout: str

# 现在
proc = subprocess.run(cmd, capture_output=True)  # 默认 text=False
# stdout: bytes, 手动 _decode_output_bytes
```

`text=False` 拿原始 bytes → 多编码 fallback 才能生效。`text=True` 把 decode 锁定到 utf-8 + 用户指定 errors, fallback 没法插手。

### 6.1 单元测试 (`tests/unit/tools/test_base_subprocess_errors.py`)

```python
def test_gbk_chinese_falls_back_to_gbk():
    """GBK 中文: utf-8 strict 失败, gbk 严格解码成功, 0 U+FFFD."""
    gbk_bytes = "看到这个图片就是压缩包的密码：\r\nbV1g6t5wZDJif^J7".encode("gbk")
    text = _decode_output_bytes(gbk_bytes)
    assert text == "看到这个图片就是压缩包的密码：\r\nbV1g6t5wZDJif^J7"
    assert "\ufffd" not in text  # 0 个替代字符
```

✅ 通过。

def test_run_subprocess_handles_gbk_output():

## 7. 兼容性与风险

### 7.1 兼容性

- `_run_subprocess` 签名不变: `(cmd, *, timeout) -> (exit_code, stdout, stderr, duration_ms)` ✅
- 返回类型不变: `(int, str, str, int)` ✅
- 子类 adapter 完全不需要改 ✅

### 7.2 风险

| 风险 | 等级 | 缓解 |
|---|---|---|
| 候选编码顺序调整导致某些 GBK 双字节字符被错误解读 | 🟢 低 | strict 解码返回不抛异常, latin-1 兜底 |
| GBK 解读 random binary 输出"假中文" (e.g. `\xa5\x68` → `\ue66e`) | 🟢 低 | gbk 严格解码, 真 random binary 大概率 utf-8/gbk 都 fail → latin-1 兜底 |
| latin-1 永远 0 U+FFFD 导致 random binary 走 latin-1 而不是 gbk | 🟡 中 | 阶段 1 strict 顺序: utf-8 → gbk → gb18030 → ... → latin-1, latin-1 永远 success 但只在前面都失败才用 |

### 7.3 向后兼容

- commit d500d79 的 `errors='replace'` 副作用被替换: 不再 U+FFFD, 而是按字符匹配选最优编码
- 任何依赖 "输出含 U+FFFD" 的下游代码 (如果有) 需要修
