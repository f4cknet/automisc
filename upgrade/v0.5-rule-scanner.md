## 2. 实施

### 2.1 新建 `core/utils/rule_scanner.py`（独立规则库）

| 规则 | category | severity | 阈值 |
|---|---|---|---|
| 敏感关键词 | `sensitive_keyword` | **5** | 出现 key/flag/secret/ctf/password |
| Binary 串 | `binary` | 4 | 仅 0/1 chars, ≥ 32 |
| Hex 串 | `hex` | 4 | 仅 0-9a-f, ≥ 32, 偶数长度 |
| Base32 串 | `base32` | 4 | A-Z 2-7, ≥ 12 chars, ≤ 6 = padding |
| Base64 串 | `base64` | 4 | A-Za-z0-9+/, ≥ 12 chars, ≤ 2 = padding, 可解码 |

**主入口**：`classify_text(text) -> list[TextMatch]`
- 单行：整段判定 (priority: sensitive > binary > hex > base32 > base64) → 不命中再子串匹配
- 多行：逐行扫 (strings 输出场景) → 整行编码记 1 match; 整行含 keyword 记所有 keyword 位置

**关键 insight**：
- **hex 是 base64 字符集子集** — 必须先检 hex 避免把 `deadbeefcafe...` 误报成 base64
- **sensitive_keyword 不在整段 token 判定** — 会触发子串匹配误判整段 ("the password is: X" 不应让整段都是 severity=5)
- **多行 vs 单行不同策略** — strings 输出多行时，整段语义无意义，要逐行

### 2.2 `encoding_detector.py` 改为薄包装

旧 `score_text_severity` / `is_*` 全部委托给 `core.utils.rule_scanner`。LSB router 33 单测无破坏。

### 2.3 `tools/shared/strings.py` 集成 rule_scanner

替换 `scan_output_for_suspicious` 旧路径 → 用 rule_scanner 逐行扫：
- 命中 `base64/base32/hex/binary` → severity=4 + suggested_action 提示 **"hex/binary -> ascii 转换（v0.5+ 迭代计划中）"** / **"base64 可试 `automisc decode base64-image --file <txt>`"**
- 命中 `sensitive_keyword` → severity=5
