## 1. 问题

**CTF 真实场景**：
- 拖 meihuai.jpg → auto-run 跑 strings → 命中一行 **35000 字符的 hex 串**（典型：嵌入文件 magic + raw bytes 十六进制表示）
- v0.5-truncate-output 设计 "截断 120 字符打印"——**但**:
  - 120 字符看不到 magic number (89504e47 占 8 chars)
  - 35000 字符根本复制不动 (CTF 赛场每秒钟都宝贵)
  - "想看原文"是废话——你根本不想看 35000 字符

**Owner 期望**：
- **长 hex 串 (>= 200 chars)**: auto-run 不要再让 user 抄（CTF 宝贵时间）→ **自动 DAG 路由** (探测 magic + 写文件 + 调 zbar/unzip 等后续)
- **短 hex 串 (< 200 chars)**: 仍打印到 output 区 (人手抄走 hex-ascii 流程)——per Owner "低于 200 字符的 hex, 你打印给我"

## 2. 实施

### 2.1 新 `core/actions/hex_router.py` (v0.5-hex-router)

```
输入: hex_text (字符串, >= 200 chars, 全 0-9a-f 字符)
  ↓
is_long_hex_text(text) -> bool
  ↓
bytes.fromhex(text) -> raw bytes
  ↓
detect_magic(raw) -> (label, ext, file_type, follow_up_desc)
  ↓
text_based_output_path(suffix=f".{ext}", purpose=f"hex_router_{file_type}")
  -> /tmp/automisc_text_outputs/hex_router_<type>_<ts>_<rand>.<ext>
  ↓
write_bytes(raw) -> 写文件
  ↓
[follow_up=True 时]:
  - image -> zbarimg 扫
  - archive (zip) -> unzip 提取
  - 其它: 仅写文件, 留给 user 手动
  ↓
返回 HexRouterResult (magic / ext / output_path / follow_up_stdout / stderr)
```

### 2.2 strings adapter 集成 (v0.5-truncate-output 升级)

**改动**：
- `SuspiciousPoint.matched_pattern` 字段——**hex 类别保留完整 m.value (不限 120 截断)**；其他类别仍 120 截断
- `_render_output` 遍历 suspicious:
  - 短 hex (< 200 chars) → 仍打印 `L<line>: <前 200 字符>`
  - 长 hex (>= 200 chars) → **不打印实际内容**：
    - 调用 `route_hex_to_file()` 自动 trigger
    - 渲染版加 `--- v0.5-hex-router: N 个超长 hex 串已自动 route ---`
    - 命中行显示 `L<line>: <hex_router 已自动处理, 见上方 summary>` (占位符, 不打印 35000 字符)

**sp.category 匹配**:
- 中文 label: "十六进制串_line1"  
- 双判断: `"十六进制" in sp.category or "hex" in sp.category.lower()` 兼容
