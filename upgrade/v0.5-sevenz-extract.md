# v0.5-sevenz-extract — 7z 解压 adapter (GUI 工具栏)

## 背景

per Owner 2026-06-20 19:48 拍板,参照 writeup <https://www.cnblogs.com/junlebao/p/13837046.html>:

> 7z x flag.vmdk -o/　　解压flag.vmdk
> 然后出现四个目录,在自己账户下,不在桌面上

实战场景:做 `面具下的flag` 这道 CTF,foremost 雕出伪加密 zip,修完后解出
`flag.vmdk`,**7z 才能解压**(.vmdk 是 VMDK 磁盘镜像格式,p7zip 17.05 支持)。

**当前架构 gap**:
- `sevenz` adapter 只 `7z l` (list) + `7z t` (test),**不解压**(per v0.5-philosophy-rethink
  owner 决策 1:探测归探测,操作归操作,auto_run 纯探测)
- `unzip` adapter 也只 `unzip -l`,不解压(真解压走 chain / 工具栏 bruteforce)
- **GUI 工具栏缺 7z 解压入口** — 用户想解 vmdk / 7z / wim / ova 等只能切 CLI

## 目标

新建一个 `sevenz_extract` adapter:
1. 真正执行 `7z x` 解压到 input 同目录 (per v0.5-output-samedir 铁律)
2. 输出 `<input_stem>__7z_extracted/` 目录(用 `extract_dir_for` helper)
3. GUI 工具栏 "Misc/Archive" 下新增一项 "📦 7z 解压",跟 sevenz / unzip / john / zip_classify 同一级
4. 不动现有 sevenz adapter(向后兼容:auto_run / chain 还在用它探测)
5. 不动现有 unzip adapter(对称:探测归探测,解压归解压)

## 设计选择 (方案 B vs 方案 A)

**方案 A**:改 sevenz adapter 加解压模式 — **否决**
- 破坏现有"快速探测"语义(主线程跑,GUi 卡)
- auto_run / chain 用 sevenz 做探测,改后行为不一致
- 探测 + 解压合一 = 慢路径污染快路径

**方案 B**:新建 sevenz_extract adapter — **采用** ✅
- 跟 unzip 对称(unzip 也只 list 不解压,真解压走 chain)
- 探测 / 解压职责单一,后续 sevenz_password_extract 也好组织
- 写盘路径清晰:`<stem>__7z_extracted/`

## 7z 解压命令 (per writeup)

```bash
7z x -y -o<output_dir> <archive>
```

参数说明:
- `x` — extract with full paths(保留目录结构)
- `-y` — assume Yes on all queries(避免交互,适合自动化)
- `-o<dir>` — output directory(**注意**: `-o` 跟路径**直接连**,无空格,这是 7z 风格)
- `<archive>` — 待解压文件

writeup 实测:
```bash
7z x flag.vmdk -o/        # writeup 用了根目录,但 automisc 走 v0.5-output-samedir 铁律,
                           # 解到 <input_stem>__7z_extracted/ 而不是根目录
```
