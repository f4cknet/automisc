# v0.5-decoder-friendly-candidate — extract_base_candidate 加 decoder-friendly 特例

## 背景

per Owner 2026-06-20 21:17 实战反馈:

```
+++++ +++++ [->++ +++++ +++<] >++.+ +++++ .<+++ [->-- -<]>- -.+++ +++.<

++++[ ->+++ +<]>+ +++.< +++++ +[->- ----- <]>-- ----- --.<+ +++[- >----

<]>-- ----- .<+++ [->++ +<]>+ +++++ .<+++ +[->- ---<] >-.<+ +++++ [->++

++++< ]>+++ +++.< +++++ [->-- ---<] >---- -.+++ .<+++ [->-- -<]>- ----- .<

=== Decoder: brainfuck ===

[!] input 区为空, 没东西可解.
提示: 在 input 区粘贴 hex 串 (e.g. '28372c37290a') 然后再点菜单 brainfuck。
```

owner 把 BF 代码直接 paste 到 input 区, 但 GUI 说"input 区为空, 没东西可解"。

## 根因

`extract_base_candidate(strict=True)` 现在的 `looks_like_base` 判定不含 BF/Ook 字符:

| 候选类型 | 现有判定 | 是否通过? |
|---|---|---|
| base64 | 含 `+/` 或行尾 `=` | ✅ |
| hex | 全 `[0-9a-fA-F]` | ✅ |
| binary | 全 `[01]` | ✅ |
| caesar 密文 | 全 `[A-Z]{4,}` | ✅ |
| **brainfuck** | 含 `+` `-` `>` `<` `[` `]` `.` `,` | ❌ 不通过! |
| **Ook!** | 含 `Ook[.!?]` | ❌ 不通过! |

`+++` `-` `<` `>` `[` `]` `.` 这些字符不在 base 字符集, 也不在 base64/hex/binary 特征里, 所以 raw BF 代码被判不算候选 → strict 模式返回 None → "input 区为空"。

## 目标

`looks_like_base` 加 2 个 decoder-friendly 特例:

1. **brainfuck 候选**: 含 BF 8 字符 `+ - < > [ ] . ,` 之一 且 长度 ≥ 12 → 算候选
2. **Ook! 候选**: 含 `Ook.` / `Ook!` / `Ook?` 任一 token 且 长度 ≥ 20 → 算候选

不动 base64/hex/binary/caesar 现有判定 (向后兼容)。
