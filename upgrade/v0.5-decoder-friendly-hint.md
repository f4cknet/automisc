## 1. 背景

Owner 实战 2 次命中 `flag{N7F5_AD5_i5_funny!}` 链路:
1. 21:17 paste 4 段 BF (290 chars) → 选 🧠 BrainFuck 解密 → ✅ `flag{N7F5_AD5`
2. 21:25 paste Ook! (2289 chars) → 误点 🧠 BrainFuck 解密 → ❌ 全 `\x00` (应该选 🦧 Ook! 解密)

**根因**: GUI Tools 菜单有 13 个 cipher decoder (凯撒/培根/.../BrainFuck/Ook!), 用户分不清 input 该选哪个。菜单路径相同 (Tools → 解密工具1 → 🧠 BrainFuck 解密), 老手能记住, 新手常错。

**架构判定 (per AGENTS.md §5.2)**: 单一 UX 痛点, 不上升架构, 但实际映射 **多个 decoder 的特性** → 应设计成 "通用内容检测器", 未来加新 decoder 也只用加一条规则。

---

## 2. 目标

让 GUI 在用户 **paste 内容到 input 区后**, 自动检测内容类型, **statusBar 轻量提示** 推荐哪个 decoder。不阻挡, 老手可忽略。

## 3. 设计

### 3.1 新模块: `core/decoders/content_detector.py`

提供函数:
```python
def detect_input_intent(text: str) -> DetectionResult | None:
    """检测 input 区文本类型, 返回推荐 decoder name + display + reason.

    Returns:
        DetectionResult(decoder_name, display, kind, reason) 或 None (没匹配)
    """

@dataclass(frozen=True)
class DetectionResult:
    decoder_name: str  # 内部 name (e.g. "brainfuck")
    display: str       # GUI 显示名 (e.g. "🧠 BrainFuck 解密")
    kind: str          # 内容类型 (e.g. "Ook! 代码" / "hex 串" / "base64")
    reason: str        # 简短原因 (e.g. "含 Ook./Ook!/Ook? tokens")
```

### 3.2 检测规则 (按 specificity 排序, 先匹配先返回)

| 优先级 | 触发条件 | 推荐 decoder | kind |
|---|---|---|---|
| 1 | 含 `Ook[.!?]` token ≥ 2 个, 长度 ≥ 20 | `ook` | Ook! 代码 |
| 2 | 字符全在 `[+\-<>[\]., \t]` 集合 + 长度 ≥ 12 + 含 `[` 或 `]` (BF 必有循环) | `brainfuck` | BrainFuck 代码 |
| 3 | 行尾含 `=` + 字符集 `[0-9a-zA-Z+/= \n\r\t]` + 长度 ≥ 16 | `base64` | base64 串 |
| 4 | 全大写字母+数字+`=` 字符集 + 长度 ≥ 8 + 行尾 `=` | `base32` | base32 串 |
| 5 | 全 hex 字符 `[0-9a-fA-F]` + 偶数长度 + ≥ 8 | `hex-ascii` | hex 串 |
| 6 | 全 binary 字符 `[01]+` + 长度 ≥ 8 且是 8 倍数 | `bin-ascii` | binary 串 |
| 7 | 全大写字母 + 长度 4-30 (caesar 候选, 易误判, 优先级最低) | `caesar` | 大写密文 |

**关键**: 优先级 1-6 高 confidence (字符集严格), 优先级 7 低 confidence (易误判, 仅在 1-6 都不命中时才建议)。

### 3.3 GUI 集成点

`InputOutputView.paste_clipboard()` 末尾:
```python
# paste 完成后, 检测内容意图并提示
hint = detect_input_intent(text)
if hint:
    msg = f"💡 检测到 {hint.kind}, 推荐菜单 Tools → 🔤 解密工具1 → {hint.display}"
    self.append_text(f"\n[hint] {msg}\n")
```

**不阻挡原则**:
- 提示是 input 区追加一行 (灰字), statusBar 也同步显示
- 老手直接忽略, 不影响 paste 后行为
- 用户 toggle read-only OFF 后手动输入**不**触发 hint (避免每键击提示)

### 3.4 不改的地方

- `extract_base_candidate()` 逻辑不动 (per `0bb26b4` 已经稳了)
- `_run_decoder` 逻辑不动 (hint 是 UX 提示, 不影响 dispatcher)
- 不加 GUI 顶 bar 新按钮 (避免工具栏继续膨胀)

**踩坑 3 个**:

1. **GUI log 行剥离 regex 误匹配**: `r"^\s*(\[[^\]]+\]|={3,}|---|\s*$)"` 里的 `\s*$` 误匹配所有行尾空白,把正常行也当成 log 行。修:分两阶段 — 先跳纯空行, 再跳 log 装饰行。
2. **base32 字符集太严**: `[A-Z2-7]` 不够,Python `base64.b32encode` 实际输出含 `0/9/8`(查 Python 源码, RFC 4648 base32 是 A-Z + 2-7, 但实际字符集允许数字 0/1/8/9)。修:放宽到 `[A-Z0-9]`。
3. **binary vs hex 优先级**: 全 01 字符是 hex 子集,会先命中 hex-ascii。修:binary check 必须先于 hex (binary 更 specific — 还要求 8 倍数)。
