## 1. 背景

### 1.1 Owner 报告 (2026-06-20 11:49 + 12:02)

> "我觉得我的 auto_run 设计存在巨大的漏洞... 不应该将所有类型的题目附件都叠砌到 auto_run 中，这样只会让程序逻辑写的越来越死... 下一步让做题人自己判断应该使用什么工具，不要过于自动化解题。"

### 1.2 真正哲学 (per STRUCTURE.md §1, 但之前没落地)

- **工具 = 找可疑点** (探测)
- **journal = 可疑点累积** (结果池)
- **做题人 = 决策下一步** (人工判断, **不抢 flag**)

之前 v0.5-DAG-chain 路线走偏 — 把"链"做成"自动解题", 违背半自动化哲学。

## 2. 新设计 (per owner 决策 C)

### 2.1 `auto_run` 改名 + 重构

**旧**: `auto_run` (跑 top 5 推荐工具 + 触发链 — **抢 flag**)
**新**: `find_suspicious_from_<type>` (跑**纯探测**工具 + **不**触发链)

**type 类型**:
- `picture` (PNG/JPG/JPEG/BMP/GIF)
- `traffic` (pcap/pcapng/cap)
- `archive` (zip/7z/rar/tar/gz)
- `binary` (bin/exe/dll/elf)

**每类工具池** (只探测, **不**雕不修不爆):

| type | 工具 |
|---|---|
| picture | zsteg / steghide / exiftool / binwalk (探测) / strings / file |
| traffic | pcap_protocol_router / tshark / strings / file |
| archive | sevenz / unzip (列表 `-l`, 不解压) / file |
| binary | file / strings / binwalk (探测) / exiftool |

### 2.2 关键边界

| 入口 | 行为 | 哲学 |
|---|---|---|
| `find_suspicious_from_<type>` (auto_run rename) | 纯探测, 写 journal | **不抢 flag** |
| GUI 工具栏 foremost / binwalk 按钮 | 雕文件 | **人工触发** OK |
| GUI 工具栏 fix_pseudo_zip 按钮 | 修复伪加密 | **人工触发** OK |
| GUI 工具栏 bruteforce_zip 按钮 | 爆破密码 | **人工触发** OK |
| `automisc chain --chain zip` (CLI) | 链触发 | **人工触发** OK |
| `automisc chain --chain foremost` (CLI) | 链触发 | **人工触发** OK |
