# v0.5-fix-find-suspicious-race-condition — 强 kill 旧 runner subprocess, 修拖新文件时旧工具段残留

> **status**: 🔄 in-progress (2026-06-29 23:00 Owner 22:57 拍板 A: 强 kill 旧 runner)
> **触发**: Owner 2026-06-29 22:30 拖 jpg 跑 picture pool (含 steghide 30s timeout) → 立即拖 zip → output 区"5 个 archive 段被压到顶部, steghide (auto FAIL) 段在底部"误读
> **根因**: `MainWindow._on_new_file_selected` line 474-486 `runner.wait(2000)` 强等 2s 强制返回, 旧 FindSuspiciousRunner 的 steghide subprocess (subprocess.run 同步阻塞) 继续在后台跑, 30s timeout 后 emit `tool_finished` 写新 output 区
> **fix**: runner.wait() 改 runner.kill() (强 terminate 嵌套 subprocess + wait 真停) + 加 subprocess handle 引用让 kill 能 terminate

---

## 1. 背景

### 1.1 实战触发 (2026-06-29 22:30)

Owner 拖 jpg → picture pool 跑 6 工具, steghide 卡 30s timeout → Owner 立即拖 zip:
1. `_on_new_file_selected` 调 `runner.stop()` + `runner.wait(2000)` 强返回
2. 旧 steghide subprocess **继续**在后台跑 (subprocess.run 同步阻塞, stop flag 不响应)
3. main thread 立刻 clear output + clear journal + 启新 archive pool runner
4. archive pool 5 工具 < 1s 跑完, 写 5 个 `=== xxx ===` 段 (在 output 区顶部)
5. **30s 后** 旧 steghide subprocess timeout 完成 → emit `tool_finished("steghide", ...)` → main thread 调 `_on_auto_tool_finished` 写 `=== steghide (auto FAIL) ===` 段 (在 output 区底部)
6. Owner 看 output 区, 滚到底部看到 steghide 段, 误以为是当前文件

### 1.2 v0.5-journal-clear-on-new-file 不解决此问题

`v0.5-journal-clear-on-new-file.md` 修的是 **journal_panel 累积** (拖新文件时清空), 但 **output_view 仍有 race condition** — 旧 steghide 段在 archive pool 段**之后**写入 output 区, Owner 仍会看到旧段误读.

### 1.3 根因 (per AGENTS §1 铁律 4 验证)

`_on_new_file_selected` line 474-486:
```python
for runner in (self._find_suspicious_runner, ...):
    if runner and runner.isRunning():
        try:
            runner.stop()         # 只设 _stopped = True
        except Exception:
            pass
        runner.wait(2000)        # 等 2s 强返回, 但 subprocess 还在跑
```

`runner.stop()` 只设 `_stopped = True` 标志, `FindSuspiciousRunner.run` line 437 在**下一个工具开始前**检查 `_stopped`:
```python
for i, tool_name in enumerate(tool_names):
    if self._stopped:            # 这里检查
        break
    ...
    result = self.core.run_tool(tool_name, self.file_path)  # 但当前工具在跑, 已 stuck 在 subprocess.run
```

**当前工具** (e.g. steghide) 仍 stuck 在 `subprocess.run` 30s, 不会被 stop flag 中断. `wait(2000)` 等 2s 强返回, 旧 subprocess 继续在 background 跑.

### 1.4 决策点 (Owner 2026-06-29 22:57 拍板 A: 强 kill 旧 runner subprocess)

> 1.A. 强 kill 旧 runner subprocess (推荐)

修法 3 步:
1. **adapter 加 subprocess handle 引用**: `core.run_tool` 调 `_run_subprocess` 时持有 `subprocess.Popen` (替代 `subprocess.run`), 写到 `self._current_proc`
2. **adapter 加 kill() 方法**: `terminate()` + `kill()` 嵌套 subprocess
3. **main_window 调 runner.kill()**: 替代 `runner.wait(2000)`, `runner.kill() + runner.wait()` 强停

## 2. 改动

### 2.1 核心改动: adapter 持有 subprocess handle (15 行)

**`src/automisc/tools/base.py`** — `_run_subprocess` 改用 `subprocess.Popen` 持有 handle, 加 `_current_proc` 字段:

```python
class ToolAdapter(ABC):
    name: str = ""
    # ...
    _current_proc: subprocess.Popen | None = None  # 持有当前 subprocess handle (kill 用)

    def _run_subprocess(self, cmd: list[str], *, timeout: float | None = None) -> tuple[int, str, str, int]:
        # 旧 subprocess 还卡着 → 强 terminate (避免 race)
        self._terminate_current_proc()

        effective_timeout = timeout if timeout is not None else self.default_timeout
        start = time.monotonic()
        # ... env 准备 ...
        try:
            proc = subprocess.Popen(
                cmd,
                stdout=subprocess.PIPE,
                stderr=subprocess.PIPE,
                env=env,
            )
            self._current_proc = proc  # 持有 handle
            try:
                stdout_bytes, stderr_bytes = proc.communicate(timeout=effective_timeout)
            except subprocess.TimeoutExpired:
                proc.kill()  # 强 kill 嵌套 subprocess
                proc.wait()
                # ... 返回 124 timeout ...
            self._current_proc = None
            # ... decode + return ...
        except FileNotFoundError as e:
            # ...
        finally:
            self._current_proc = None

    def _terminate_current_proc(self) -> None:
        """强 kill 当前 adapter 持有的嵌套 subprocess (if any)."""
        if self._current_proc and self._current_proc.poll() is None:
            try:
                self._current_proc.terminate()
                self._current_proc.wait(timeout=1.0)
            except Exception:
                try:
                    self._current_proc.kill()
                except Exception:
                    pass
        self._current_proc = None
```

**`src/automisc/tools/paths.py`** — **不**改 (paths.py 不涉及 subprocess)

**`src/automisc/core/orchestrator.py`** — `run_tool` 调 adapter.run 前先调 `adapter._terminate_current_proc()` (重入保护 — adapter 跑新工具前确保旧工具已停):

```python
def run_tool(self, tool_name: str, file_path: str) -> ToolResult:
    cls = get_tool_class(tool_name)
    adapter: ToolAdapter = cls()
    adapter._terminate_current_proc()  # 重入保护: 旧 adapter 实例可能复用, 先清
    result = adapter.run(file_path)
    # ...
```

**注**: `ToolAdapter` 单例复用 (per v0.1.1 设计), `_terminate_current_proc` 是 idempotent (没 _current_proc 啥也不做).

### 2.2 main_window 调 runner.kill() (5 行)

**`src/automisc/gui/main_window.py`** line 474-486 改:

```python
# 旧 (line 474-486):
for runner in (self._find_suspicious_runner, ...):
    if runner and runner.isRunning():
        try:
            runner.stop()
        except Exception:
            pass
        runner.wait(2000)

# 新:
for runner in (self._find_suspicious_runner, ...):
    if runner and runner.isRunning():
        try:
            # v0.5-fix-find-suspicious-race-condition: 强 kill 嵌套 subprocess
            if hasattr(runner, "kill_subprocess"):
                runner.kill_subprocess()
            else:
                runner.stop()
        except Exception:
            pass
        runner.wait(2000)  # wait 仍 2s, kill 之后 2s 内 wait 必返回
```

**`src/automisc/gui/auto_runner.py`** — `FindSuspiciousRunner` 加 `kill_subprocess()` 方法, `AutoRunner` 同 (向后兼容):

```python
class FindSuspiciousRunner(QThread):
    # ... existing code ...

    def kill_subprocess(self) -> None:
        """v0.5-fix-find-suspicious-race-condition: 强 kill 当前 tool 的 subprocess."""
        self._stopped = True
        # 调当前 tool adapter 的 _terminate_current_proc
        # 注意: runner 不持有 adapter 实例 (adapter 跑一次就丢), 改用 core.orchestrator
        if hasattr(self, "core") and self.core is not None:
            # core 单例, 它持有最近一次 adapter 实例? 实际 core 不持有
            # 简化: runner 通过 tool_name 调 current_proc
            pass
```

**更简单** — adapter 的 `_terminate_current_proc` 是**实例方法**, 但 adapter 实例跑完一个工具就丢 (`cls()` 新建). **要解决**必须让 orchestrator 持有 adapter 实例, **或**让 adapter 改成 class-level static method (单例).

**修正修法**: orchestrator 持有最近一次 adapter 实例:

```python
class CoreOrchestrator:
    def __init__(self, ...):
        self._last_adapter: ToolAdapter | None = None

    def run_tool(self, tool_name, file_path) -> ToolResult:
        cls = get_tool_class(tool_name)
        adapter = cls()
        self._last_adapter = adapter  # 持有
        result = adapter.run(file_path)
        return result

    def kill_last_subprocess(self) -> None:
        """强 kill 最近一次工具的嵌套 subprocess (auto-run 切换用)."""
        if self._last_adapter:
            self._last_adapter._terminate_current_proc()
            self._last_adapter = None
```

main_window 调 `core.kill_last_subprocess()` (替代 `runner.kill_subprocess()`).

## 5. 风险

| 风险 | 概率 | 缓解 |
|---|---|---|
| subprocess.Popen 比 subprocess.run 略慢 (startup ~10ms) | 低 | CTF 工具调用是 0.1s-30s 级, 10ms 无关 |
| 子进程终止有 race (Windows process tree 不跟 kill) | 中 | `terminate()` → `wait(timeout=1.0)` → `kill()` fallback |
| adapter 实例复用 (per v0.1.1 design) 导致 _current_proc 残留 | 低 | `run_tool` 入口先 `_terminate_current_proc()` idempotent |
| 7z / foremost 等 Windows GUI 工具 `terminate()` 不能优雅退出 | 中 | `terminate()` 给 1s 优雅, 然后 `kill()` 强杀 |
| 现有测试依赖 subprocess.run 行为 (Popen 行为略不同) | 低 | 现有测试不读 Popen, 只读 returncode/stdout/stderr, Popen 兼容 |
