## 2. automisc 实测（per §5.3 step 1）

### 2.1 auto-run journal（steghide 超时）

```
=== steghide (auto FAIL) ===
[stderr] subprocess timeout after 30.0s
exit_code: 124 | suspicious_points (0):
```

**关键观察**:
- steghide 在 `FIND_SUSPICIOUS_PICTURE_TOOLS` 第 2 位（紧跟 lsb_tool）
- `steghide info` 命令本身正常执行，但某种情况下被 `default_timeout=30.0`（per `tools/base.py:90`）杀掉
- exit_code 124 是 subprocess timeout 信号（与 timeout CLI 工具一致，per `tools/base.py:160`）
- 30s 噪音让前面快速工具都没机会跑完，journal 体验差

### 2.2 根因诊断

1. **顺序问题**：steghide 在第 2 位
   - 当前：`lsb_tool / steghide[1] / exiftool / binwalk / strings / file`
   - steghide 超时 → 后面的 exiftool / binwalk / strings / file 都按顺序排队等
   - 5 工具 × 30s = 150s 潜在总耗时

2. **steghide 工具特性**：
   - 是密码学 stego 工具（需要空密码或猜密码）
   - 比 lsb_tool / exiftool / binwalk 等"无密码"工具慢得多
   - 实战经验：steghide 通常作为最后 fallback 工具

3. **steghide 在 auto-run 池的合理性**：
   - 应该保留在 auto-run 池（提供基础探测能力，per v0.5-stegseek-remove 已替换 stegseek）
   - 但**顺序**应放最后
