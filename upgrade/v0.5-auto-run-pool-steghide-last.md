## 1. 背景

### 1.1 问题

**当前 `FIND_SUSPICIOUS_PICTURE_TOOLS` 顺序**（per `src/automisc/gui/auto_runner.py:59-66`）:

```python
FIND_SUSPICIOUS_PICTURE_TOOLS = [
    "lsb_tool",     # 1
    "steghide",     # 2  ← 30s 慢路径密码学 stego
    "exiftool",     # 3
    "binwalk",      # 4
    "strings",      # 5
    "file",         # 6
]
```

**Owner 实战 journal**（per v0.5-train-015）:

```
=== steghide (auto FAIL) ===
[stderr] subprocess timeout after 30.0s
exit_code: 124 | suspicious_points (0):
```

### 1.2 根因诊断

1. **顺序问题**: steghide 在第 2 位
   - steghide 超时 → 后面的 exiftool / binwalk / strings / file 都按顺序排队等
   - 5 工具 × 30s 潜在总耗时 = 150s 噪音
   - 快速工具（lsb_tool 几秒）应该先跑出结果，steghide 作为最后 fallback

2. **steghide 工具特性**: 密码学 stego（需要空密码或猜密码），比无密码工具慢得多
   - 默认 timeout 30s（per `tools/base.py:90 default_timeout=30.0`）
   - 空密码 extract 单独有 5s timeout（per `steghide.py:240`），但 `steghide info` 用默认 30s

3. **owner 实战经验**: "通常在其他工具没发现可疑信息后才会尝试 steghide" — 这是通用 CTF 实战经验

### 1.3 历史决策参考

- **v0.5-stegseek-remove** (2026-06-28): 之前 `FIND_SUSPICIOUS_PICTURE_TOOLS` 用 stegseek，2026-06-28 重构成 steghide
  - 重构时**只换了工具名**，没动顺序
  - v0.5-stegseek-remove 没考虑顺序问题
- **v0.5-train-015**: Owner 实战触发顺序问题反馈

### 1.4 Owner 决策（per 2026-06-28 16:46）

> "把 steghide 的流程放到 auto_run 的最后一步，因为根据经验，通常在其他工具没发现可疑信息后才会尝试 steghide"

**核心决策**：
- ✅ 顺序调整（架构层，所有图都受影响）
- ❌ 不删 steghide（仍需保留 auto-run 池基础探测能力）
- ❌ 不改 steghide 内部逻辑（默认 30s timeout 不动）

## 3. 详细设计

### 3.1 `FIND_SUSPICIOUS_PICTURE_TOOLS` 顺序调整

**新顺序**（per Owner "先快速工具, 后慢路径 fallback"）:

```python
FIND_SUSPICIOUS_PICTURE_TOOLS = [
    "lsb_tool",     # 1. 快速, 替代 zsteg + 自研 (per v0.5-lsb-tool-bitplane-preview-matrix)
    "exiftool",     # 2. 快速, EXIF metadata
    "binwalk",      # 3. 探测 (不 -e), 快速
    "strings",      # 4. rule_scanner 可疑字符串, 快速
    "steghide",     # 5. 慢路径密码学 stego, 最后 fallback (per v0.5-auto-run-pool-steghide-last)
    "file",         # 6. 文件类型, 兜底
]
```

**Owner 经验映射**:
- lsb_tool → exiftool → binwalk → strings → **steghide** → file
- 前 4 个工具跑出可疑点（一般 5-10s 总耗时）→ steghide 不必跑（auto-run 已命中 SP 终止条件 per §1 铁律 7）
- 前 4 个工具没命中 → steghide 作为 fallback（30s 慢路径，可接受噪音）

### 3.2 单测验证

**`tests/unit/gui/test_auto_runner_pool_order.py`**:

```python
"""验证 FIND_SUSPICIOUS_PICTURE_TOOLS 顺序 (per v0.5-auto-run-pool-steghide-last)."""
from automisc.gui.auto_runner import FIND_SUSPICIOUS_PICTURE_TOOLS


def test_pool_order_steghide_last_before_file():
    """steghide 在 file 之前 (慢路径 fallback 顺序)."""
    assert "steghide" in FIND_SUSPICIOUS_PICTURE_TOOLS
    steghide_idx = FIND_SUSPICIOUS_PICTURE_TOOLS.index("steghide")
    file_idx = FIND_SUSPICIOUS_PICTURE_TOOLS.index("file")
    assert steghide_idx < file_idx, (
        f"steghide should be before file (fallback 顺序), "
        f"got steghide={steghide_idx}, file={file_idx}"
    )


def test_pool_order_fast_tools_first():
    """快速工具 (lsb_tool/exiftool/binwalk/strings) 在 steghide 之前."""
    fast_tools = ["lsb_tool", "exiftool", "binwalk", "strings"]
    steghide_idx = FIND_SUSPICIOUS_PICTURE_TOOLS.index("steghide")
    for tool in fast_tools:
        assert tool in FIND_SUSPICIOUS_PICTURE_TOOLS, f"{tool} should be in pool"
        tool_idx = FIND_SUSPICIOUS_PICTURE_TOOLS.index(tool)
        assert tool_idx < steghide_idx, (
            f"{tool} (fast) should be before steghide (slow fallback), "
            f"got {tool}={tool_idx}, steghide={steghide_idx}"
        )


def test_pool_size_still_6():
    """池大小仍 6 (per §1 铁律 7: auto-run 池不扩张)."""
    assert len(FIND_SUSPICIOUS_PICTURE_TOOLS) == 6
```

## 4. 风险评估

| 风险 | 等级 | 缓解 |
|---|---|---|
| steghide 放最后后, 某些依赖 steghide "先跑" 的逻辑断 | 🟢 低 | steghide 只在 auto-run 跑, 单图独立 SP 决定;前 4 个工具未命中时 steghide 仍跑 |
| 单测断言 `steghide < file` 但未来扩展需要 steghide 在 file 之后 | 🟢 低 | 单测只断言相对顺序, 不锁死 pool size;扩展时同步改单测 |
| 实战题中 steghide 应早跑才能命中 (例如 steghide 提取后用 binwalk 二次分析) | 🟢 低 | steghide auto-run 是**只探测不雕不修不爆** (per §1 铁律 7);只报 SP, 不实际 extract 文件;后续如需 extract 由 Owner 手动跑 |
