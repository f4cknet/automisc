## 3. AI 兜底（per §5.3 step 3，6 步拿 flag）

### 3.1 步骤 1：exiftool -b -Padding 抽 Padding（误判路径）

```powershell
& "D:\hacktools\misc\automisc-fresh\extend-tools\bin\win-x64\exiftool.exe" -b -Padding "C:\Users\zmzsg\Downloads\梅花香之苦寒来\meihuai.jpg"
```

抽 1974 字节 → 全 0x00 + 3 个异常字节 `0x1c 0xea 0x08`（不是 flag，**是 EXIF Padding 字段填充**）。

**结论**：Padding 不是 flag 藏身处。

### 3.2 步骤 2：JPEG 末尾追加检测（关键！）

```python
data = JPG.read_bytes()
ffd9 = data.rfind(b"\xff\xd9")
tail = data[ffd9 + 2:]
# → FFD9 @ 0x52f4, 追加 649566 bytes
```

**关键发现**：JPEG 真实图片数据在 FFD9 (21236 bytes) 就结束了，**但文件有 670804 bytes** —— 后面 649566 bytes 是 JPEG 文件格式允许的"附加数据"区域（很多工具会忽略），**这是真 flag 的藏身处**。

**automisc 漏报根因**：binwalk 只看标准 magic（ZIP/PNG/7z/RAR 等），hex-encoded 坐标数据 (`28 37 2c 37 29 0a` 字面 ASCII) 没有 magic 触发；strings 只看 printable ASCII 段，hex 字面被跳过。

### 3.3 步骤 3：解析 hex-encoded 坐标

```python
# tail 是 ASCII 字符, 内容是 hex 编码的 (x,y)\n 坐标
decoded = bytes.fromhex(tail.decode("ascii"))  # 324783 bytes
text = decoded.decode("ascii")
import re
pairs = re.findall(r"\((\d+),(\d+)\)", text)   # 35019 个坐标点
# → 范围: X [7, 271], Y [7, 271], 全唯一
```

**关键发现**：35019 个坐标对，画在 272x272 网格上 = QR 码。

### 3.4 步骤 4：PIL 绘制 QR

```python
from PIL import Image
img = Image.new("L", (272 + 20, 272 + 20), 255)  # 白底 + 10px quiet zone
for x, y in coords:
    img.putpixel((x + 10, y + 10), 0)  # 黑点
img.save("qr_plum.png")
```

**坑**：第 1 次 `Image.new("1", ..., 1)` + `putpixel(0)` cv2 解码失败（缺 quiet zone + 黑白反）；第 2 次 `Image.new("L", ..., 255)` + 加 10px padding + cv2 直接解成功。

### 3.5 步骤 5：cv2 QRCodeDetector 解码

```python
import cv2, numpy as np
img = cv2.imread("qr_plum.png", cv2.IMREAD_GRAYSCALE)
data, pts, _ = cv2.QRCodeDetector().detectAndDecode(img)
print(data)  # → flag{40fc0a979f759c8892f4dc045e28b820}
```

**结果**：🎉 **`flag{40fc0a979f759c8892f4dc045e28b820}`**

## 4. automisc 应补能力（架构升级触发）

### 4.1 🚨 Bug 2: exiftool 中文乱码（架构升级触发）

**根因**：`src/automisc/tools/shared/exiftool.py` adapter 没传 `-charset utf8`，exiftool 按 OS locale（Windows GBK code page）解码 EXIF Unicode 字段，UTF-8 字节被当 GBK 解码 → 乱码（`å›¾ç©·flagè§`）。

**修复**（adapter 1 行改动）：`cmd = [self.binary, "-charset", "utf8", ...]`

**架构判定**：Win + 中文 EXIF 必然触发（中文 Windows 系统 metadata 全乱）→ **≥3 道同类命中** → 架构升级 ✅

### 4.2 🚨 Bug 1: stegseek Win 无版（架构升级触发）

**根因**：stegseek 是 Linux-only C++ 工具，GitHub 无 Windows release。`automisc-gui` auto-run 仍触发 stegseek → exit_code=127 干扰 journal。

**修复**：
- GUI 菜单 `stegseek` 标 `✗` unavailable（v0.5-platform-extend-tools 已预期但 GUI marker 还没全实现）
- `FIND_SUSPICIOUS_PICTURE_TOOLS` auto-run 池**移除 stegseek**（同 zsteg Windows unavailable，per v0.5-lsb-detector）
- 同样的 unavailable 工具白名单：`steghide`（无 Windows 版）/ `stegseek`（Linux-only）/ `zsteg`（Ruby gem）

**架构判定**：Win 必然触发（≥3 道同类）→ 架构升级 ✅

### 4.3 🚨 Bug 3: file Win 无原生版（架构升级触发）

**根因**：`file` 是 Unix CLI，Win 无原生版。`automisc-gui` auto-run 触发 file → exit_code=127 干扰 journal。

**修复**：2 选 1：
- **A. 装 [magic-win](https://github.com/nscaife/file-windows) prebuilt `file.exe` 到 `extend-tools/bin/win-x64/`**（1 binary ~3MB，跟 exiftool 风格一致，**推荐**）
- B. `file` adapter 走 `python-magic-bin`（PyPI 包，跨平台，免 binary，但破坏 Win binary 一致性）

**架构判定**：Win 必然触发（≥3 道同类）→ 架构升级 ✅

### 4.4 automisc 漏报分析（次要，不动代码）

- binwalk 不识别 JPEG 末尾追加的 hex-encoded 坐标数据 → **不修**，因为：
  - JPEG 末尾追加是 JPEG 标准允许但不规范
  - 检测成本高（要扫所有非标准 trailing data）
  - 这类题型（QR 坐标）**目前只有 1 次实战命中**（N<3，按 §5.2 不升架构）
- strings 漏 hex-encoded 字符串 → **不修**，同理
- auto-run 命中 G/B 通道异常但没真 flag → **当前设计如此**（铁律 7：auto-run 不雕不修不爆，只写 SP），属于预期行为
