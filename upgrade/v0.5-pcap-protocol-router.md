## 1. 背景

### 1.1 当前行为（v0.5-DAG-chain 后）

`Challenge/greatescape.pcap` 跑通流程：
1. `automisc run --tool tshark --file greatescape.pcap` → exit 0
2. 47 个 suspicious_points 全是 severity=1（关键字 "Cipher" / "Password" 协议字段被通用扫描误判）
3. 0 个 severity≥3 的命中
4. **期望行为**：识别"913 个 TLS 包 + 1 个 FTP-DATA 流" → 高亮"发现 TLS 加密 + 候选 key 文件" → 提示用户去 Wireshark 手动配 `--ssl.keys`

### 1.2 协议层分布（pcap 实际数据）

| 协议 | 包数 | 字节 | 占比 | 角色 |
|---|---|---|---|---|
| TLS | 913 | 621,804 | 66.5% | HTTPS / FTPS / SMTPS 加密层 |
| HTTP | 46 | 30,901 | 3.3% | TLS 解密后才会出现的明文 |
| FTP | 16 | 1,828 | 0.2% | **明文**控制连接（藏 key 文件名 + 凭据） |
| FTP-DATA | 1 | 3,341 | 0.4% | **明文**数据传输（ssc.key 文件本体） |
| SMTP | 33 | 6,375 | 0.7% | 邮件（明文或 STARTTLS）|
| OCSP | 42 | 28,072 | 3.0% | TLS 证书状态查询 |

→ **明文辅助协议（FTP/SMTP）** 跟 **加密主协议（TLS）** 共存，**这就是 TLS 密钥可能藏的地方**。

### 1.3 Owner 提的关键 insight

> "通过协议分析，你会发现大量的TLS协议，大概率是找tls解密钥匙，那就去非加密流量中找解密的key,通常解密密钥是.key，或者是.pub结尾的，如果有其他tls密钥的可疑文件后缀你也可以加进来。发现可以密钥文件，那接下来的事情就由人工在wireshark里操作了"

→ **automisc 不应尝试自动解密**（凭据敏感 + 需用户授权）—— 但应**自动发现 key 候选**并**提示用户去 Wireshark 操作**。

## 2. 设计决策（Owner Q&A）

### Q1: 哪些协议算"非加密辅助协议"？

**Owner 答**（基于 greatescape 实测）：
- **明文辅助协议**（key 候选藏身处）：FTP / FTP-DATA / HTTP（80）/ SMTP（25）/ Telnet（23）/ TFTP（69）/ SNMP（161）
- **加密主协议**（key 用来解这些）：TLS / HTTPS（443）/ FTPS（990）/ SMTPS（465/587）
- **判定标准**：协议是否默认加密（依赖 `tshark -d` 解码表 + 端口号）

### Q2: key 候选文件怎么识别？

**Owner 答**：
- **后缀白名单**（首版）：`.key` / `.pub` / `.pem` / `.crt` / `.cer` / `.der` / `.p12` / `.pfx` / `.pkcs8` / `.openssh`
- **FTP STOR/RETR 提取文件名**：从 FTP 控制流里 grep `STOR xxx` / `RETR xxx`
- **HTTP 路径**：从 HTTP request URI 提取 `/xxx.key` 等
- **SMTP 附件**：暂不实现（ATTACH 处理复杂，v0.5+ 后续迭代）
- **非文本流量传输**：FTP-DATA 流的 raw bytes 写到 `/tmp/`，**等用户授权后再判定是不是 key**

### Q3: 是否自动解密 TLS？

**Owner 答**：**不**。
- 凭据敏感（per 上次 `mavis-trash` key 落盘事件，工具已主动 block 凭据落盘）
- Wireshark 手动 `--ssl.keys` 操作简单，automisc 不抢这一步
- automisc 只负责**找到 key 候选 + 提示命令模板**

### Q4: auto-run 怎么改？

**Owner 答**：
- FileRouter 加新分支：`pcap + TLS占比 > 30%` → 推荐 `pcap-protocol-router` 工具链
- 工具链输出 2 类信息：
  1. **协议占比饼图**（文字版）：`TLS 66% / HTTP 3% / FTP 0.2% / ...`
  2. **key 候选清单**：每个候选一个 SuspiciousPoint，severity=4（不是 5，因为还要用户确认）

### 3.1 新增模块

```

  - is_cipher_protocol(name) → bool   # TLS/HTTPS/FTPS/...

  - KeyFinder 类
    - scan_ftp_stor(pcap) → list[str]    # 从 FTP STOR 命令提取文件名
    - scan_http_uri(pcap) → list[str]    # 从 HTTP URI 提取 .key/.pub/.pem 等
    - scan_ftp_data_streams(pcap) → list[dict]  # 提取 FTP-DATA 流到候选
    - has_tls_packets(pcap) → bool       # 是否含 TLS 流量
    - extract_suspicious_files(...) → list[SuspiciousPoint]

src/automisc/core/chains.py
  + build_pcap_protocol_router_chain(file_path): DAG
    1. ProtocolClassifyAction: tshark -q -z io,phs → 协议占比
    2. KeyFinderAction: 扫 FTP STOR / HTTP URI / FTP-DATA → key 候选
    3. （如果命中 key 候选）WiresharkHintAction: 打印 --ssl.keys 命令模板
    4. DAG 终止条件: 协议层发现完 → 停（不再尝试自动解密）
```

### 3.2 行为表

| 场景 | 协议分布 | KeyFinder 输出 | SuspiciousPoint | 行为 |
|---|---|---|---|---|
| 全 HTTPS（无 FTP） | TLS 99% / HTTP 0% | 空 | 无 | 输出"全加密，无明文辅助" |
| HTTPS + FTP（greatescape） | TLS 66% / FTP 0.2% / HTTP 3% | `[ssc.key]`（FTP STOR） | 1 个，severity=4 | 输出 Wireshark 命令模板 + 链停 |
| HTTPS + HTTP（明文） | TLS 80% / HTTP 20% | `[/login.php, /admin.key]` | N 个，severity=4 | 同样输出命令模板 |
| 全 HTTP（明文） | HTTP 100% | （不算 key 候选） | 0 | 走老路径：HTTP flag 扫描 |
| 全 SMTP | SMTP 100% | （附件暂不支持）| 0 | 走老路径：SMTP 邮件头 |

### 3.3 决策树（Owner 最终版）

```
PcapProtocolRouterAction.run(pcap_path)
  1. tshark -q -z io,phs pcap_path → protocols dict
  2. protocols_pretty_print(protocols)  # 打印协议占比
  3. if not has_tls_packets(protocols):
       → "未发现 TLS 加密流量，链停（无 key 候选可找）"
       → severity=0
  4. if has_plaintext_auxiliary(protocols):
       → KeyFinder.scan(pcap_path)
       → 输出 key 候选列表
       → for each 候选:
            SuspiciousPoint(category="tls_key_candidate", severity=4,
                            suggested_action="用 Wireshark 配 --ssl.keys <key_path> 重抓 + 搜 http contains FLAG")
  5. 打印 Wireshark 手动命令模板（per Q3 不自动解密）：
     tshark -r <pcap> -o "tls.keys_list:<ip>,<port>,http,<key_path>" \
       -Y "http contains FLAG" -T fields -e http.request.uri -e http.file_data
  6. 链停
```

## 4. 实现细节

### 4.1 协议分类（tshark 输出解析）

```python
# tshark -q -z io,phs 输出示例:
#   frame                                    frames:2756 bytes:935469
#     ip                                   frames:2756 bytes:935469
#       tcp                                frames:2756 bytes:935469
#         tls                              frames:913 bytes:621804
#         ftp                              frames:16 bytes:1828
#         ...

import re
_PROTO_LINE_RE = re.compile(r"^\s+(\w+)\s+frames:(\d+)\s+bytes:(\d+)\s*$")

def parse_io_phs(stdout: str) -> dict[str, tuple[int, int]]:
    """返回 {协议名: (frames, bytes)}"""
    result = {}
    for line in stdout.splitlines():
        m = _PROTO_LINE_RE.match(line)
        if m:
            proto, frames, bytes_ = m.group(1), int(m.group(2)), int(m.group(3))
            # 过滤低层（frame/ip/tcp/udp）
            if proto.lower() in ("frame", "eth", "ip", "tcp", "udp", "sll"):
                continue
            result[proto] = (frames, bytes_)
    return result
```

### 4.2 KeyFinder 后缀白名单（per Owner Q2）

```python
# HTTP URI 提取
_HTTP_URI_RE = re.compile(r"GET\s+(/[^\s]+\.(?:key|pub|pem|crt|cer|der|p12|pfx|pkcs8|openssh))",

```

## 6. 风险 & 缓解

| 风险 | 缓解 |
|---|---|
| mavis-trash 误吞 key 落盘事件 | **不**在 automisc 内写 key 文件（Q3）；只输出 Wireshark 命令模板，让用户自己 extract |
| SuspiciousPoint 误报高（severity=4 但实际不是 key） | 后缀白名单 + FTP STOR 双向校验（控制流 + 数据流大小匹配）|
| tshark `-z io,phs` 大 pcap 慢 | 限 max_frames=5000（per 现有 tshark adapter 的 `-c 1000` 模式）|
| SMTP 附件支持 v0.5+ | 本迭代**不**做；att v0.6+ 后续 |
| TLS 1.3 密钥（pre-master-secret）模式 | 本迭代只支持 RSA private key；pre-master-secret log 走 v0.5+ 后续（per ctf-forensics network.md）|
