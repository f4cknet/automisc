## 2. Owner 反馈原文 (2026-06-20 10:46)

> "00000038.zip 有一部分数据是伪加密，一部分是真加密。你应该按照 ctfwiki 的中 zip 伪加密的原理进行算法调整，不要认为全部数据都是伪加密，也不要因为第一个数据区是真加密就停止对后面的数据区的伪加密判断。能理解我说的吗？"

**Owner 真实诉求**：
1. **逐 entry 独立判断**（per-entry classification）— 不应一锅端判"全 zip 伪加密"或"全 zip 真加密"
2. **不 short-circuit**（per entry）— 遇到第一个真加密 entry **不**停，继续判断后面 entry
3. 修：**只**清**伪加密** entry 的加密位，**不**动**真加密** entry（修真加密位是无效操作——data 真加密修了也白修）

---

## 3. automisc 现状 bug（per 反思）

### 3.1 算法层 bug — `_is_pseudo_encrypted` 不够精细

当前实现是「任一 entry 是伪加密 → True」——**粗粒度**判断。**没有**「哪些 entry 是伪加密 vs 真加密 vs 明文」的细粒度分类。

```python
# 现有 (粗粒度 bool 返回)
def _is_pseudo_encrypted(zip_path: Path) -> bool:
    """任一 entry 是伪加密 → True"""
    ...
```

**问题**：
- 调用方**不知道**是哪些 entry 伪加密
- 修复时只能"按 flag bit 0 全部清"——这会**误修真加密** entry 的加密位（虽然修了也白修，但破坏性修复）

### 3.2 修复层 bug — `FixPseudoEncryptionAction` 修所有 flag bit 0

```python
# 现有 (粗粒度修复)
for lfh_off in lfh_offsets:  # 扫**所有**外层 LFH
    flag = fixed[lfh_off + 6] | (fixed[lfh_off + 7] << 8)
    if flag & 0x1:
        fixed[lfh_off + 6] = fixed[lfh_off + 6] & 0xFE
        fixed_count += 1
```

**问题**：
- 不区分 entry 是伪加密还是真加密
- **真加密 entry 也被清**了加密位（修了也白修——data 真加密 zipfile 仍报 fail）

### 3.3 owner 真实样本特殊性

`00000038.zip` 的 3 个外层 entry：
- `asd/` (目录, LFH/CDH bit0=0) → **clear**
- `asd/good-已合并.jpg` (LFH=0/CDH=1, data 明文 deflate) → **pseudo** ✅
- `asd/qwe.zip` (LFH=0/CDH=0, store 透明) → **clear**

**外层 0 个真加密 entry**——所以现有代码**误修真加密**的问题在 owner 真实样本上**不显现**。但**算法层**仍然不严谨，**有混合 zip 时会出错**。

## 5. 应补能力（架构升级输入）

| 能力 | 现状 | 应补 |
|---|---|---|
| **`_classify_zip_entries` 函数** | 无 | 抽出来 per-entry 分类: 返回 `{pseudo: {...}, real: {...}, clear: {...}}` (含 lfh_offset/cdh_offset 用于精确修复) |
| **`_is_pseudo_encrypted` 升级** | 粗粒度 bool | 内部用 `_classify_zip_entries`，返回 `len(pseudo) > 0`（保持 bool 向后兼容）|
| **`FixPseudoEncryptionAction` per-entry 修** | 修所有 flag bit 0 | 用 `_classify_zip_entries` 拿 pseudo set，**只**清这些 entry 的 LFH + 对应 CDH bit 0 |
| **`TryUnzipAction` fail 行为** | 整体 fail + is_pseudo | 保持（已经是 per-entry 解，bad entry 跳过）|
| **per-entry 分类的 clear 边界** | 无 | `comp_size < 12` → pseudo（太短不可能有真加密 header）；`data[11] in range(12)` → real；否则 pseudo |
| **混合 zip 单测** | 无 fixture | 加 1 pseudo + 1 real + 1 clear 的混合 fixture（Owner 真实样本是更特殊的 case）|

## 7. 待 Owner 拍板的决策点

### 决策 1: 修复边界 — 真加密 entry 修真加密位还是完全不动?

| 选项 | 优点 | 缺点 |
|---|---|---|
| A. 只清伪加密 entry 的位（owner 建议）| 语义清晰，**不**做无效操作 | 漏修真加密 → zipfile 仍 fail（行为没差）|
| B. 修真加密 entry 也清（当前实现）| 实现简单 | 修了也白修，**破坏** zipfile 内部状态校验 |
| C. 清伪加密 + 报告真加密需要密码 | 信息最全 | 仍修真加密位（副作用）|

**我建议 A**（per owner 反馈 + 修不破坏原则）— 跟 owner 现在反馈的"不修真加密"对齐。

### 决策 2: per-entry 分类后 `TryUnzipAction` fail 行为

- 现状：整体 fail → 看 `_is_pseudo_encrypted` → True/False
- 新：应该**按 fail 时的具体 entry** 判断？或者**整体**判断？

**我建议保持**「整体判断」语义——但内部用 per-entry 分类驱动。如果 fail 时**有**伪加密 entry → `is_pseudo=True` → fix_pseudo 修 → 再 try；都没有 → `is_pseudo=False` → chain 停。
