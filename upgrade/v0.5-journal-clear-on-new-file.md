# v0.5-journal-clear-on-new-file — 拖新文件时同步清 journal_panel

> **status**: 🟡 design → 🔄 in-progress（2026-06-29 21:54 实施中）
> **触发**: Owner 2026-06-29 21:47 实战反馈 — 拖入 zip 文件看到 `=== steghide (auto FAIL) ===`，误以为是这次 auto-run 跑了 steghide
> **根因**: `output_view` 在拖新文件时清空（per v0.5-hex-ascii-fix §2.2 / v0.5-clear-on-new-file），但 `journal_panel` **不**清 — 累积了之前拖 jpg 跑过的 steghide SP，做题人看 journal 误以为是当前文件的线索
> **fix**: 1 行 — `MainWindow._on_new_file_selected` 调 `self.output_view.clear()` 之后加 `self.journal_panel.clear()`

---

## 1. 背景

### 1.1 v0.5 早期决策（per `v0.5-hex-ascii-fix.md` §5）

v0.5-hex-ascii-fix spec §5 列"后续候选"时明确：

> **v0.5-journal-clear**：journal panel 是否也清空？还是保留？目前 journal 累积 (历史审计用), 拖新文件不清

**当时 Owner 选择不清** — 理由：journal 跨文件累积做历史审计用。

### 1.2 实战触发（2026-06-29 21:47 Owner 反馈）

Owner 拖入 `C:\...\00000038.zip`（29KB，伪加密）：

```
[新文件] 00000038.zip (v0.5-clear-on-new-file 已清空旧 output)
...
[auto-run] 启动 find_suspicious_from_archive, 跑 5 个工具: ['sevenz', 'unzip', 'zip_classify', 'file', 'strings']
(纯探测, 不雕不修不爆; 做题人看 journal 决定下一步)

=== steghide (auto FAIL) ===                       <-- 误以为这次跑的
[stderr] subprocess timeout after 30.0s
exit_code: 124 | suspicious_points (0):

[auto-run done] 4/5 OK · 9 suspicious points found
出现BUG了，需要FIX。拖入zip文件，auto_run不应该跑steghide
```

**实际根因**：
- AI 跑了一遍 archive pool 5 个工具，**确实没有 steghide**（9 SP 完全对上：3+0+1+1+4）
- 但 `journal_panel` 里**累积了之前拖 jpg 时 steghide FAIL 留下的 SP**
- Owner 看 journal 看到 steghide (auto FAIL) 段，**误以为是这次 auto-run 跑的**（虽然输出格式 `=== xxx (auto FAIL) ===` 来自 output 区，但 Owner 复制时混淆 / journal_panel 跟 output 一起看）

**AI 侧排查结论**（per AGENTS §1 铁律 4 关 4 验证）：
- `pick_suspicious_pool('test.zip')` → `('archive', ['sevenz', 'unzip', 'zip_classify', 'file', 'strings'])` 5 个工具，0 steghide
- `FIND_SUSPICIOUS_ARCHIVE_TOOLS` list `src/automisc/gui/auto_runner.py:73-79` 确认无 steghide
- 37 个 `test_find_suspicious_runner.py` 测试全过（包括 archive pool 白名单 + extension 路由）
- `grep "steghide"` 全代码 174 匹配中 0 个是 `run_tool("steghide", zip)` 显式调用
- 实测跑 archive pool 5 个工具，0 steghide 输出，9 SP 跟 Owner 报告完全一致

**结论**：auto-run 路径**绝对**不会跑 steghide。steghide FAIL 是 **journal_panel 跨 session 累积** + Owner 视觉误读。

### 1.3 决策点（Owner 2026-06-29 21:53 拍板）

> "开，建议新拖入就 journal+PANEL.CLEAR"

**改 v0.5 早期决策**：从"journal 累积（历史审计）"→"拖新文件时清 journal"。

**理由**（Owner 实战 + 行业惯例）：
1. 实战证明 journal 跨文件累积**误导**而非"审计"——做题人看 journal 不知道哪条对应哪个文件
2. journal 已经有 time 列 + tool 列 + file 列（带 file_name）—— **单文件审计**已经够
3. 跨文件审计**应该走 Core journal**（`core/journal.py` 落盘 JSONL，per v0.5-roadmap.md §5），不是 GUI widget
4. 跟 `output_view.clear()` 行为对齐 —— 一个文件对应一份干净的视图

## 2. 改动

### 2.1 核心改动（1 行）

`src/automisc/gui/main_window.py:_on_new_file_selected` (line 463+)：

```python
        # 2. 清空 output (核心: per Owner 2026-06-14 "每次有新的 input 就要清空原来的 output")
        self.output_view.clear()
        # 清完留下 [cleared] 标记, 但 read-only 模式不便用户编辑 — 同时切到可编辑
        # 实际上用户拖入新文件后通常不会马上编辑, 保持 read-only 即可
        # 仍打 [cleared] 标记保留线索
        self.output_view.append_text(
            f"[新文件] {file_path.name}  (v0.5-clear-on-new-file 已清空旧 output)"
        )
```

→ 改成：

```python
        # 2. 清空 output (核心: per Owner 2026-06-14 "每次有新的 input 就要清空原来的 output")
        self.output_view.clear()
        # 2.1 清空 journal (v0.5-journal-clear-on-new-file, per Owner 2026-06-29 21:53):
        #     之前拖图跑过的 SP 残留在 journal_panel, 拖新文件时一起清, 避免跨文件误读
        self.journal_panel.clear()
        # 清完留下 [cleared] 标记, 但 read-only 模式不便用户编辑 — 同时切到可编辑
        # 实际上用户拖入新文件后通常不会马上编辑, 保持 read-only 即可
        # 仍打 [cleared] 标记保留线索
        self.output_view.append_text(
            f"[新文件] {file_path.name}  (v0.5-clear-on-new-file 已清空旧 output + journal)"
        )
```

**3 行改动**（不是 1 行）：加 `self.journal_panel.clear()` + 改 [新文件] 标记文案 + 加注释

### 2.2 同步改动

- `output_view.append_text` 文案加 " + journal" 让用户知道也清了
- 不动 `core/journal.py`（Core 落盘逻辑保留，跨 session 审计还是走 Core）
- 不动 `journal_panel.py:clear()`（line 118-119 已经存在，只调 `self.tree.clear()`，1 行实现）

### 2.3 跟 v0.5 早期决策的关系

- **推翻** v0.5-hex-ascii-fix.md §5 "v0.5-journal-clear 后续候选 / journal 累积保留" 决策
- **保留** Core journal (`core/journal.py`) 落盘 — 跨 session 审计用，per v0.5-roadmap.md §5 待办
- **新行为**: GUI `journal_panel` (QTreeWidget) 拖新文件清；Core journal (内存 + 未来落盘) 持续累积
- **理由**: GUI widget 是**当前文件**视图，Core journal 是**审计**记录 — 两个用途分离
