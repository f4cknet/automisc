## 1. 背景

### 1.1 问题陈述 (per Owner 06-21 12:18 反馈)

Owner 实测发现: CLI 能跑 `automisc chain --chain lsb-bytes --channels G --bit 0 --scan-order col --byte-bit-order MSB --file image.png`,但 **GUI 工具栏找不到 lsb-bytes 入口**。

### 1.2 根因

`v0.5-lsb-byte-stream-extract` (commit `a528f65`) 实施时:
- ✅ CLI 完整支持(`__main__.py:201` `elif chain_name == "lsb-bytes":` 分支)
- ✅ `chain_runner.py` 没注册 (5 chain 没 lsb-bytes)
- ✅ `main_window.py` `_CHAIN_NAMES = ("zip", "zip-full", "binwalk", "foremost", "lsb")` 也没加

**spec §3.2 自己写了 "GUI Run → Chain 菜单新增 lsb-bytes 入口 + 参数 dialog + Tools 菜单自动渲染 Magic Sniffer"**,但 commit 落地时 GUI 改动整段跳过。这是 spec → commit 同步失败的**已知欠账**。

### 1.3 Owner 决策

> "把这个注册到GUI中,应该是通过 adapter 调度吧?"

Owner 直觉是 adapter 调度,实际**不是** —— `lsb-bytes` 是 **DAG chain** 类型(`build_lsb_bytes_chain(channels, bit, scan_order, byte_bit_order) -> DAG`),由 2 个 Action 串联:
- `binwalk_extract` (链头)
- `LSBBytesExtractAction` (4 参数 user-controlled)

GUI 走 **ChainRunner** 路径(`_run_chain` → `ChainRunner.run` → `dag.execute(context)`),不是 adapter 也**不是**单 action。

详见 STRUCTURE.md §3.5 架构辨析。

### 1.4 与现有 5 个 chain 的关键差异

| 维度 | 现有 5 chain (zip/zip-full/binwalk/foremost/lsb) | lsb-bytes |
|---|---|---|
| 参数 | **无参数** (调用 builder 无参) | **4 参数** (channels/bit/scan_order/byte_bit_order) |
| chain_runner 调用 | `dag = chain_builders[name]()` | `dag = chain_builders[name](channels=..., bit=..., ...)` |
| GUI 入口 | 菜单项直接调 `_run_chain(name)` | 菜单项调 `_run_chain(name, extra_context={4 个参数})` |
| GUI 交互 | 点一下就跑 | **弹 modal QDialog** 收 4 参数再跑 |

**这是 GUI 第一个带参数的 chain**,所以要新建 `LSBBytesParamDialog(QDialog)`。

## 2. 目标

### 2.1 能力 F: lsb-bytes 加到 GUI 工具栏

**目标**: 在 GUI 提供 lsb-bytes chain 入口 + 4 参数 dialog 收参,跟 CLI 行为完全一致。

### 2.2 行为契约 (per LSBBytesExtractAction.run line 196-260)

`LSBBytesExtractAction.run()` 已实现 context override 逻辑:

```python
channels = _parse_channels(context.get("channels") or context.get("__lsb_channels__") or self.channels)
bit = int(context.get("bit") or context.get("__lsb_bit__") or self.bit)
scan_order = context.get("scan_order") or context.get("__lsb_scan_order__") or self.scan_order
```

**context 优先 key**: `__lsb_channels__` / `__lsb_bit__` / `__lsb_scan_order__` / `__lsb_byte_bit_order__` (带 `__` 前缀,跟 v0.5-steghide-GUI 的 `__wordlist__` / `__password__` 风格一致)

GUI dialog 收集 4 参数 → `extra_context` → ChainRunner → `chain_builders["lsb-bytes"]` 收参建 DAG → DAG 透传 context 到 `LSBBytesExtractAction.run` → override 默认值。

### 3.1 模块改动

```

  + _CHAIN_NAMES 加 "lsb-bytes"  (新位置: lsb 之后)
  + _run_chain 增加 lsb-bytes 分支: 弹 LSBBytesParamDialog → 收 4 参数 → 走 _run_chain(extra_context={4 参数})

src/automisc/gui/lsb_bytes_dialog.py  ← 新
  + class LSBBytesParamDialog(QDialog):
      - 4 个控件 (QComboBox × 3 + QSpinBox × 1):
        · channels: QComboBox ["R", "G", "B", "A", "RG", "RB", "GB", "RGB", "RGBA"] (per _VALID_CHANNELS)
        · bit:      QSpinBox 0..7 (默认 0)
        · scan_order: QComboBox ["row", "col"] (默认 row)
        · byte_bit_order: QComboBox ["MSB", "LSB"] (默认 MSB)
      - "Preset" 按钮 (QPushButton) → 弹 preset menu:
        · "N=NP 默认 (G 通道 bit 0 col MSB)"  ← Owner 实战用
        · "全通道默认 (RGB bit 0 row MSB)"    ← spec §1 默认值
        · "自定义"  (保持当前 4 dropdown)
      - get_kwargs() -> dict 收 __lsb_channels__/__lsb_bit__/__lsb_scan_order__/__lsb_byte_bit_order__
      - OK / Cancel 按钮
```

**为什么 QComboBox for channels 而不是 QListWidget**:
- 单选场景 (一次跑一个 channel combo),QComboBox 更简洁
- 预设 "G 通道" 对应 QComboBox 一个 item,不需要多选 UI
- per `_parse_channels` 实现,字符串 "G"/"RGB"/"RGBA" 都支持,QComboBox items 列 8 个常用组合

**为什么 preset 按钮**:
- N=NP 题实战用 "G 通道 bit 0 col MSB" (4 个非默认参数),手动选 4 dropdown 太繁琐
- preset 直接一键填好,实战友好

### 3.3 渲染流程 (跨文件追踪)

```
1. GUI 启动 → main_window._build_menubar()
2. 遍历 _CHAIN_NAMES = (..., "lsb", "lsb-bytes") → QAction("Run &lsb-bytes chain")
3. QAction.triggered → lambda: self._run_chain("lsb-bytes")
4. _run_chain 检测 chain_name == "lsb-bytes" → 弹 LSBBytesParamDialog.exec_()
5. dialog.get_kwargs() → {"__lsb_channels__": ..., "__lsb_bit__": ..., ...}
6. ChainRunner(chain_name="lsb-bytes", extra_context={4 个 key})
7. ChainRunner.run: chain_builders["lsb-bytes"](channels=..., bit=..., scan_order=..., byte_bit_order=...)
8. DAG.execute(context) → LSBBytesExtractAction.run(context)
9. Action.run 读 context.__lsb_channels__ → 覆盖 self.channels 默认值
```

## 7. 待 Owner 拍板的决策点

### Q1: dialog 4 个控件用什么 widget?

| 选项 | channels | bit | scan_order | byte_bit_order | 优点 | 缺点 |
|---|---|---|---|---|---|---|
| **A** | QComboBox | QSpinBox | QComboBox | QComboBox | 单选简洁,符合实战 | 不能多选 (但实战都是单选, OK) |
| **B** | QListWidget (多选) | QSpinBox | QRadioButton | QRadioButton | 灵活,可同时勾多通道 | UI 复杂,实战用不上多通道 |

**Mavis 建议 A** — 实战单选场景,QComboBox 简洁;多通道场景已经预设了 "RGB" / "RGBA" 组合,不需要多选 UI。

### Q2: 默认参数 preset 按钮要不要?

| 选项 | preset 菜单 | 优点 | 缺点 |
|---|---|---|---|
| **A** | "N=NP 默认 (G 通道 bit 0 col MSB)" + "全通道默认 (RGB bit 0 row MSB)" + "自定义" | 实战友好, N=NP 题一键填 | 多 1 个按钮 |
| **B** | 不要 preset,用户自己选 4 dropdown | 简单, 不挡 dropdown 视线 | N=NP 题实战要选 4 次,繁琐 |

**Mavis 建议 A** — Owner 06-21 实战触发需求就是这个,N=NP 默认一键填能省 80% 实战时间。

### Q3: GUI 入口位置?

| 选项 | 位置 | 优点 | 缺点 |
|---|---|---|---|
| **A** | Run → Chain 菜单 (跟 zip/lsb 等并列) | 跟其他 chain 一致,符合 spec §3.2 原方案 | Chain 菜单变长 |
| **B** | Tools 工具栏 → 🔍 Magic Sniffer 旁边,新建 "🎯 LSB 字节流" 分类 | 跟 magic_sniffer 配套显示 | 跟 chain 概念错位 (这是 chain 不是单工具) |

**Mavis 建议 A** — `lsb-bytes` 是 chain,跟 zip/lsb 并列在 Chain 菜单最一致;Tools 工具栏是给 adapter/decoder 用的。

### Q4 (待 Owner 决定, Mavis 默认): chain_runner 4 参数用什么 key 前缀?

我倾向 `__lsb_channels__` / `__lsb_bit__` / `__lsb_scan_order__` / `__lsb_byte_bit_order__` (跟 v0.5-steghide-GUI 的 `__wordlist__` / `__password__` 风格一致,带 `__` 前缀避免污染 user context)。

## 8. 风险 + 缓解

| 风险 | 缓解 |
|---|---|
| ChainRunner 改 chain_builders 加 lsb-bytes 破坏其他 chain | 现有 5 chain 无参调用,新 lsb-bytes 走独立分支 (`if chain_name == "lsb-bytes": build_lsb_bytes_chain(**kwargs)`) |
| LSBBytesParamDialog 阻塞主线程 | 用 `QDialog.exec()` modal (非 QThread, dialog 只阻塞当前窗口) |
| preset 按钮 N=NP 默认填错参数 | 单测覆盖 preset get_kwargs() 返回值,跟 writeup 字面参数 (G 通道 bit 0 col MSB) 严格一致 |
| context key 命名冲突 (e.g. `__lsb_bit__` 撞到其他 chain) | 带 `__lsb_` 前缀,跟现有 `__wordlist__` / `__password__` 风格一致,避开通用 key |
| GUI dialog 跟 chain_runner 之间 context 序列化失败 (extra_context 是 dict) | 4 个 key 都是基本类型 (str/int),无序列化风险 |
| regression 现有 5 chain | 单测覆盖 `chain_builders["zip"]()` / `chain_builders["lsb"]()` 无参调用没变 |
